## 一、项目概述

### 1.1 赛制背景

- **参赛规模**：5 支队伍，每队 4 人
- **五大环节**：个人必答 → 团队共答 → 擂台抢答 → 识图找茬 → 服务飞花令
- **计分结构**：基本盘（封顶 50 分）+ 浮动环节（无上限·可负）= 总分不封顶，最高者胜
- **预计用时**：约 105 分钟

### 1.2 实现现状

重写已完成。系统由两个页面 + 一个共享状态模块构成，围绕五大环节的赛制实现：

| 能力 | 状态 | 实现位置 |
|------|------|---------|
| 五环节状态机（`currentRound` / `roundPhase`） | ✅ | `shared.js` `initR1()`~`initR5()` |
| 七种题型判分 | ✅ | `shared.js` 各 `scoreRx()` |
| 抢答（含读题期违规扣分、补抢） | ✅ | `shared.js` `r3TryBuzz()` / `r3EarlyBuzz()` / `r3OpenSupplement()` |
| TTS 题目朗读 + 语音报分（双引擎） | ✅ | `shared.js` TTS 模块 + 可选的 `tts_server/`，见 12.11 |
| 倒计时（跨窗口无漂移） | ✅ | `shared.js` 计时器模块（`startTimer()`~`getRemainingMs()`） |
| 超时自动判定 | ✅ | `index.html` `onTimerExpired()` / `r3AutoTimeout()`，见 15.4 |
| 抽签（出场顺序 / 图题分配 + 锁定 + 强制覆盖暗门） | ✅ | `shared.js` 抽签模块（`drawTeamOrder()`~`forceOverrideTeamOrder()`） |
| 翻牌选题（第 1/2/4 环节） | ✅ | `shared.js` `initCardDeck()` / `selectCard()` / `confirmFlip()` |
| 第四环节场景图上屏 | ✅ | `shared.js` `getR4ImageSrc()` + `display.html` `updateQuestion()` |
| 分环节上限控制 | ✅ | `shared.js` `applyTeamScore()` 统一入口，见第六章 |
| 并列检测 / 加赛 | ❌ | 见第十六章 |

> 历史背景（v2.0 时的起点）：重写前是一个通用抢答器，只有单一题型（文本/图片/视频）、
> 抢答 + 手动判分，无环节概念。这段历史仅供理解 storage key 为何升到 `v3`，
> 现已无代码残留。

### 1.3 技术约束

- **主体纯前端**：两个页面无构建步骤、无 npm 依赖，双击即运行。
  v3.1 新增的 `tts_server/`（Python）是**可选**组件：不启动则语音回退浏览器原生，
  比赛流程完全不受影响。除它之外没有任何服务端依赖。
- **跨窗口同步**：控制台（`index.html`）与大屏（`display.html`）通过 BroadcastChannel
  实时广播 + localStorage 持久化双通道同步。
- **无框架**：Vanilla JS，CSS 自定义属性，不引入第三方库。
- **兼容性**：现代 Chromium 内核浏览器（Chrome 90+）。
  原生语音依赖 `speechSynthesis`；本地 TTS 服务依赖 `fetch` + `<audio>`（见 12.11）。
- **页面角色标记**：`index.html` 在加载 `shared.js` 前设置 `window.IS_CONTROL = true`
  （`index.html:155`，紧邻 `<script src="shared.js">` 之前）。TTS 与计时器启动
  **仅在控制台侧执行**，大屏不发声，避免两个窗口同时朗读。
  `isTTSAvailable()` 以此为前置判断。

#### 运行方式

题库通过设置面板的文件选择框手动导入（`index.html` `loadQFile()`），代码中**没有**
`fetch('./questions.json')` 之类的自动加载，因此双击 HTML 用 `file://` 打开不会触发
CORS 问题，可直接运行。

> ⚠️ **改代码后必须用静态服务器调试**。Chrome 对 `file://` 的子资源（`shared.js`）
> 缓存极顽固，改完刷新甚至硬刷新（Ctrl+Shift+R）都可能仍在跑旧代码，会让人误判问题：

```bash
cd C:/Users/admin/Desktop/quiz_buzzer
python -m http.server 8080     # 然后访问 http://localhost:8080/index.html
```

> ⚠️ **注意**：`file://` 与 `http://localhost:8080` 的 **localStorage 是两套独立存储**。
> 在 8080 下调试看到的题库/分数与双击打开时不是同一份，别混淆。

状态持久化在 localStorage 的 `rz_contest_v3` 键下，刷新不丢；改动状态结构后如需清空，
在控制台执行 `localStorage.removeItem('rz_contest_v3')` 再刷新。

#### 状态结构升级的兼容性

`load()` 用 `deepMerge(defaultState(), loaded)` 合并，**新增字段会自动回填默认值**，
旧存档不会因为缺字段而报错（已实测：v3.0 的存档在 v3.1 下能正确补出 `tts.engine`、
`tts.serverUrl`、`r4.imageFiles`，且队名/分数/题库全部保留）。

反过来，**已删除的字段会作为垃圾残留**在旧存档里（如 v3.0 的 `timer.autoExpire`），
因为 `deepMerge` 会原样拷贝 `loaded` 里的未知键。这些残留无害——没有代码读它们，
且 `startTimer()` 整体重建 `timer` 对象时就会消失。

### 2.1 三个核心文件的分工

| 文件 | 职责 | 不做什么 |
|------|------|---------|
| `shared.js` | 状态定义、持久化、跨窗口广播、全部业务动作函数、TTS、计时器、抽签、翻牌 | 不碰 DOM |
| `index.html` | 主控台 UI（6 个面板 P0–P5）、事件绑定、调用 `shared.js` 动作 | 不写业务规则（`onTimerExpired()` 是例外，见 15.4） |
| `display.html` | 大屏渲染（题目/场景图/分数/翻牌/抽签/飞花令），只读 state | 不修改 state，不发声 |
| `tts_server/` | 【可选】高音质中文语音合成 | 不感知赛制，只做「文本 → 音频」 |

> ⚠️ **`index.html` 定义了一个与 `shared.js` 同名的 `stopTimer()`**，内容一致
> （都是 `state.timer.state = 'idle'`）。因 `shared.js` 先加载，控制台内的调用实际走
> `index.html` 的那一份。行为无差异，但属于重复定义，见 16.3 第 26 项。

### 2.2 题库构建链路

`questions.json` **不是手写的**，由 `parse_qbank.py` 从 `raw_qbank.txt` 解析生成：

```
荣资商厦服务技能大赛整合题库.docx
        │ （人工提取文本）
        ▼
   raw_qbank.txt
        │ python parse_qbank.py
        ▼

**改题库时应改 `raw_qbank.txt` 后重跑脚本，而不是直接手改 `questions.json`**，否则下次
重跑会覆盖手改内容。脚本内的路径是硬编码绝对路径（`C:/Users/admin/Desktop/quiz_buzzer/`），
迁移目录时需同步修改。

## 三、数据模型

### 3.1 队伍与成员

```js
// 队伍对象
{
  id: Number,          // 1~5，固定 ID
  name: String,        // "一队"~"五队"，可自定义
  color: String,       // 颜色（CSS 色值），展示用
  members: String[],   // 长度 4，成员姓名
  scores: {            // 各环节得分，key 为 ROUND_KEY
    r1: Number,        // 第一环节（个人必答）队伍合计
    r2: Number,        // 第二环节（团队共答）队伍合计
    r3: Number,        // 第三环节（擂台抢答）可正可负
    r4: Number,        // 第四环节（识图找茬）
    r5: Number,        // 第五环节（飞花令）
  },
  memberScores: {      // 个人必答环节 · 成员得分（key 为成员 index 0~3）
    0: Number,
    1: Number,
    2: Number,
    3: Number,
  }
}
```

### 3.2 题库结构

题库是一个扁平数组，**不按环节分组**，靠每题的 `round` 字段区分环节：

```js
{ "questions": [ /* Question[] */ ] }
```

**没有统一的题目对象**——不同题型的字段集差异很大，只有 4 个字段是全体共有的：

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | String | 唯一 ID，格式 `r{环节}_{类型缩写}_{序号}`，如 `r1_001` / `r3_tf_001` |
| `round` | Number | 所属环节 1–5 |
| `type` | String | 题型枚举（见 3.3） |
| `stem` | String | **题干文本**（注意：不是 `content`） |

其余字段按题型出现，各题型的完整字段见 **第七章 · 题库 JSON 格式规范**。

> ⚠️ **易踩的坑**：

- 题干字段是 **`stem`**，不是 `content`。代码全线读 `q.stem`
  （如 `shared.js` `buildQuestionSegments()`）。
- 分值字段是 **`score_correct` / `score_wrong`**（下划线命名），不是 `scorePerPoint`。
- `blanks` 是**对象数组** `[{hint, answer[]}]`，不是空数（Number）。
- **不存在** `title` / `content` / `media` / `spotCount` / `usedBy` / `flyingFlowerTheme`
  这些字段，代码里也没有任何地方读它们。
- `score_wrong` **只有第三环节的 216 道题有**（R3 全部为 -2）。R1/R2/R4 题目没有此字段，
  因为这些环节答错不扣分。
- `theme` 题（3 道）**没有任何分值字段**，飞花令的 +1 / +3 由代码规则决定。
- `spot` 题有 `score_correct: 1`，但 **`scoreR4()` 并不读它**——找茬得分是按勾选点数
  硬编码每处 +1 计算的。该字段目前是装饰性的。

### 3.3 题型枚举

代码与题库中实际存在 **7 种**题型（`index.html` `renderQBox()` 内的 `typeMap` 是权威列表）：

| 枚举值 | 中文名 | 使用环节 | 题量 | 有 `options` | 答案字段 |
|--------|-------|---------|------|-------------|---------|
| `single` | 单选 | 1、3 | 40 + 16 | ✅ | `answer: "B"` |
| `multi` | 多选 | 2、3 | 2 + 67 | ✅ | `answer: ["A","B","C"]` |
| `judge` | **判断** | 3 | 65 | ❌ | `answer: "√"` 或 `"×"` |
| `fill` | 填空（单空） | 3 | 68 | ❌ | `answer: "45"` |
| `fill_multi` | 多空填空 | 2 | 2 | ❌ | `blanks: [{hint, answer[]}]` |
| `spot` | 识图找茬 | 4 | 5 | ❌ | `spots: [{key, label}]` |
| `theme` | 服务飞花令 | 5 | 3 | ❌ | `answerPool: String[]` |

合计 268 题。

> ⚠️ **与 v2.0 设计稿的命名差异**（设计稿的名字在代码中一律不存在）：

| v2.0 设计稿 | 实际实现 |
|------------|---------|
| `fill_single` | `fill` |
| `spot_image` | `spot` |
| `flying_flower` | `theme` |
| （未设计） | **`judge`** ← 判断题，65 道，占第三环节近三成 |

`judge` 是设计稿完全遗漏、但代码与题库都已实现的题型。它没有 `options`，
`answer` 直接存 `√` / `×` 字符；控制台在 `index.html` `renderQBox()` 中把它
与 `single`/`multi` 一同当作可交互选择题渲染，选项由 UI 侧提供而非题库提供。

### 3.4 全局状态对象（state）

```js
const state = {
  // --- 基础配置 ---
  teams: Team[5],            // 5 支队伍
  keymap: { [teamId]: String[] }, // 抢答器键位映射
  logo: String | null,       // 品牌 Logo（base64）
  brandName: String,         // 品牌名称

  // --- 题库 ---
  questions: Question[],     // 全量题库

  // --- 环节状态机 ---
  currentRound: Number,      // 当前环节 1~5（0=赛前）
  roundPhase: String,        // 'idle'|'running'|'judging'|'finished'

  // --- 第一环节专用 ---
  r1: {
    // 出场顺序直接读取 draw.teamOrder，不单独存储
    currentTeamIdx: Number,   // 当前第几支队（0~4，指向 draw.teamOrder 的下标）
    currentMemberIdx: Number, // 当前队内第几名选手（0~3）
    currentQIdx: Number|null, // 当前题在 questions[] 中的索引（全局索引，非环节内序号）
    usedQIds: String[],       // 已用题目 id，防重复出题
    timerSec: Number,         // 单题限时（默认 15）
  },
  // 注：无 qPerPerson 字段。"每人 2 题"由主持人操作流程保证，代码不强制。

  // --- 第二环节专用 ---
  r2: {
    currentQIdx: Number|null, // 当前题在 questions[] 中的索引
    usedQIds: String[],       // 已用题目 id
    timerSec: Number,         // 单题限时（默认 40）
    answers: { [teamId]: String[] }, // 各队当题作答
  },

  // --- 第三环节专用（抢答）---
  r3: {
    currentQIdx: Number|null,
    usedQIds: String[],       // 已用题目 id
    timerSec: Number,         // 单题限时（默认 15）
    buzzState: String,        // 'idle'|'reading'|'armed'|'locked'（共 4 个状态）
                              // reading：TTS 正在读题，按抢答器=违规
                              // armed：大屏显示"开始抢答"，可合法抢答
                              // locked：某队抢到，等待答题
                              // 注：补抢不是独立状态，r3OpenSupplement() 把状态设回
                              // 'armed' 并置 supplementUsed=true（原 'supplement'
                              // 死枚举已于 2026-07-16 删除）
    buzzedTeam: Number | null,
    selectedTeam: Number | null,
    selectedMember: Number | null,
    buzzPulse: Number,        // 自增，触发展示页抢答成功动画
    lastBuzzTeam: Number | null,
    supplementUsed: Boolean,  // 当题补抢是否已用
    currentReadText: String,  // 当前正在朗读的题目文本（违规后重读用）
  },

  // --- 第四环节专用 ---
  r4: {
    currentTeamIdx: Number,   // 当前第几支队（指向 draw.teamOrder 的下标）
    currentQIdx: Number|null, // 当前图题在 questions[] 中的索引
    usedQIds: String[],
    timerSec: Number,         // 单图限时（默认 60）
    // 各点位判定。key = spot.key（题库里的 "a1"/"b2" 等字符串，非数字序号）
    spotJudge: { [spotKey]: Boolean },
    extraSpots: { desc: String, ts: Number }[], // 评委现场认定的额外找茬点
    // imageKey → 图片文件路径（2026-07-16 新增，默认 images/图A.png ~ images/图E.png，
    // 可用 setR4ImageFile(imageKey, path) 覆盖；大屏经 getR4ImageSrc() 读取）
    imageFiles: { [imageKey]: String },
  },

  // --- 第五环节专用 ---
  r5: {
    currentThemeIdx: Number, // 当前令题序号（0~2）
    teamOrder: Number[],     // 轮次顺序，initR5() 时从 draw.teamOrder 复制后独立
    currentTurnIdx: Number,  // 当前轮到第几位（指向 r5.teamOrder 的下标，跳过已出局队）
    activeTeams: Number[],   // 当前令题中尚未出局的队伍（team.id）
    timerSec: Number,        // 单次限时（默认 10）
    usedAnswers: { teamId, answer, ts }[], // 本令题已用答案（防重复）
    themeWinners: { themeIdx, teamId }[],  // 各令题擂主记录（累积，非单值）
  },

  // --- 翻牌选题 ---
  cardFlip: {
    enabled: Boolean,          // 总开关（默认 true）
    // 各环节是否启用翻牌（可独立关闭）；第 2 环节也已接入
    rounds: { 1: true, 2: true, 4: true },
    // 当前牌组（每次 initCardDeck() 时重新生成）
    cards: [
      // 第 1/2 环节：{ id, cardNum, qId, revealed, used }
      // 第 4 环节：  { id, cardNum, teamId, imageKey, revealed, used }
    ],
    // 当前正在等待翻牌的环节和上下文

#### 状态恢复的合并策略

`load()` 用 `deepMerge(defaultState(), loaded)` 恢复状态（`shared.js` `load()`），而非直接
覆盖。这意味着**新增 state 字段时老存档能自动补上默认值**，无需清 localStorage。但注意
`deepMerge` 只递归普通对象，**数组是整体替换**——所以往数组类字段加默认元素不会生效。

反向的代价：`loaded` 里的未知键会被原样保留，**已删除的字段会在旧存档里残留**
（详见 1.3「状态结构升级的兼容性」）。

### 3.5 判分事件（ScoreEvent）

全部判分流水追加进 `state.history`，由 `shared.js` `logEvent(event, skipAnnounce)`
统一写入。**事件不是统一结构**，各环节按需附加字段。

**公共字段**（各环节都有）：

```js
{
  round: Number,      // 环节 1~5
  teamId: Number,
  teamName: String,
  correct: Boolean,   // 是否判为正确（注意：v2.0 未列出此字段，实际必有）
  delta: Number,      // 实际得分变化量（已经过上限裁剪，可正/负/零）
  reason: String,     // 见下表
  ts: Number,         // 时间戳
}
```

**按环节附加的字段**：

| 附加字段 | 类型 | 出现在 | 说明 |
|---------|------|-------|------|
| `memberIdx` / `memberName` | Number / String | R1、R3 | 个人得分归属；队伍整体得分（R2/R4/R5）**不带这两个字段** |
| `memberDelta` | Number | R1 | **个人**实际入账。未封顶答对时等于 `delta`；封顶时 `delta=0` 而它仍为题目分值；**违规时 `delta=-2` 而它为 0**（违规只扣队伍分） |
| `capped` | Boolean | R1 | 本次是否触发上限裁剪 |
| `note` | String | R1（违规）、任意 | 文字说明（违规缘由等）。**仅在非空时才附加该字段** |
| `qId` | String | R1、R2、R3、R4 | 题目 id |
| `fromBuzz` | Boolean | R3 | 恒为 `true`，标记来自抢答 |
| `foundCount` | Number | R4 | 实际找出的点数（裁剪前） |
| `perSpot` | Number | R4 | 每处分值（读自 `q.score_correct`，兜底 1） |
| `answer` | String | R5 | 有效答案内容 |
| `eliminated` | Boolean | R5 | 出局事件标记 |
| `manual` | Boolean | `adjust` | 手动调分标记 |

> ⚠️ **`delta` 恒表示「队伍」实际入账**。R1 封顶后 `delta = 0` 但 `memberDelta = 2.5`；
> R1 违规时 `delta = -2` 但 `memberDelta = 0`。统计个人成绩时必须读 `memberDelta`
> （或直接读 `team.memberScores`），**不能累加 `delta`**。

> ⚠️ **不存在 `note` 字段**（v2.0 设计了但未实现，代码中 0 处引用）。备注信息目前无处存放。

**`reason` 取值**：

| 取值 | 触发处 | delta |
|------|-------|-------|
| `correct` | 各环节**完全**答对 | 依题型 |
| `partial` | **R2 部分答对**（多选漏选 / 多空对了一部分） | >0 且 <满分 |
| `wrong` | 各环节答错 / R5 出局 | R3 读 `q.score_wrong`（兜底 -2）；R1/R2/R5 为 0 |
| `violation` | R1 队内外提示 `scoreR1('violation')`；R3 读题期抢跑 `r3EarlyBuzz()` | `R1_VIOLATION_PENALTY` / `R3_VIOLATION_PENALTY`（均为 -2，具名常量，不读题库） |
| `timeout` | R3 超时：`index.html` `doR3Timeout()`（手动按钮）或 `r3AutoTimeout()`（倒计时归零） | 读 `q.score_wrong`，兜底 -2 |
| `spot` | R4 结算 `scoreR4()` | 找对点数（≤10） |
| `flower_valid` | R5 有效答案 | +1 |
| `flower_winner` | R5 擂主 | +3 |
| `adjust` | 手动调分 `adjustScore()` | 任意 |

> ⚠️ **`partial` 与 `correct` 的区分**（2026-07-16 修正）：此前 R2 漏选时 `reason` 被标为
> `'correct'` 但 `correct` 属性为 `false`，导致语音播报"答对"与实际判分相悖。现在三态分明：
> 全对 → `correct`，得分但不满分 → `partial`（播报"部分答对，得X分"），零分 → `wrong`。

> ⚠️ **注意**：`timeout` 是唯一**不由 `shared.js` 生成**的 reason——它在控制台的
> `doR3Timeout()` / `r3AutoTimeout()` 里手工拼装事件对象后调 `logEvent()`。
> 业务规则外泄到了 UI 层，是已知的架构瑕疵（见 16.3 第 23 项）。

## 四、计分规则实现

### 4.1 第一环节 · 个人必答（单选）

**赛制规则**：
- 每人 2 题，单题 15 秒，口头作答
- 答对：+2.5 分（同时计入队伍分与个人分）
- 答错/超时：0 分，不扣分
- 违规（队内外提示）：本题 0 分 **且** 队伍扣 2 分 ✅ 已实现（2026-07-16）
- 单队上限 20 分（4 人 × 2 题 × 2.5 分 = 20，即全对即封顶）

**实际实现**：

```js
// result: true = 答对 | false = 答错/超时 | 'violation' = 违规
// note:   可选缘由，非空时写入 event.note
function scoreR1(result, note = '') {
  const team      = getTeamByIdx(state.r1.currentTeamIdx);  // 从 state 取，不传参
  const memberIdx = state.r1.currentMemberIdx;              // 同上
  const q         = state.questions[state.r1.currentQIdx];
  const baseScore = q?.score_correct ?? 2.5;                // 分值来自题库，兜底 2.5

  const isViolation = result === 'violation';
  const correct     = result === true;
  const delta       = correct ? baseScore
                    : (isViolation ? R1_VIOLATION_PENALTY : 0);

  // 队伍分走上限裁剪；个人分记裁剪【前】的真实得分（2026-07-16 解耦，见 6.2）
  const actual = applyTeamScore(team.id, 'r1', delta);
  // 违规扣的是【队伍】分，个人分只记该题本身得分（0），不倒扣
  const memberDelta = isViolation ? 0 : delta;
  team.memberScores[memberIdx] += memberDelta;

  logEvent({ round:1, teamId:team.id, memberIdx,
             delta: actual,        // 队伍实际入账（裁剪后 / 违规为负）
             memberDelta,          // 个人实际入账
             capped: delta !== actual,
             reason: correct ? 'correct' : (isViolation ? 'violation' : 'wrong'),
             qId: q?.id, note, ... });
}
```

> 实现位置：`shared.js` `scoreR1()`

**三态判分的调用方式**：

| 场景 | 调用 | 队伍分 | 个人分 |
|------|------|--------|--------|
| 答对 | `scoreR1(true)` | +2.5（裁剪） | +2.5（不裁剪） |
| 答错 / 超时 | `scoreR1(false)` | 0 | 0 |
| 违规 | `scoreR1('violation', '缘由')` | **-2（不受上限约束）** | 0 |

> ⚠️ **注意：违规扣的是队伍分，不扣个人分。**
>
> 赛制原文是「本题 0 分**且**队伍扣 2 分」——处罚对象是队伍。若个人分也倒扣，
> 会污染「最佳个人」的评选数据（见 6.2）。
>
> 违规扣分能在**已封顶**的队伍上生效（20 → 18），这依赖 `applyTeamScore()`
> 对负 `delta` 不做裁剪，见 6.1 的说明。

控制台的 `[违规 -2]` 按钮会先弹框要求填写缘由，写入 `ScoreEvent.note` 留痕，
并在判分流水 CSV 中导出（见 16.2 第 16 项）。缘由可留空，但取消弹框则不判分。

⚠️ 旧调用 `scoreR1(true)` / `scoreR1(false)` **行为完全不变**，无需改动既有代码。

**与设计稿的差异**：

| 设计稿 | 实际 |
|-------|------|
| `scoreR1(teamId, memberIdx, result)` 三参数 | `scoreR1(correct)` **单个布尔值**，队伍/选手/题目全部从 `state.r1` 读取 |
| `result` 三态含 `'violation'` | 只有 `true`/`false` 两态 |
| 违规 -2 分 | ❌ **未实现**——`scoreR1` 没有违规分支，第一环节违规目前只能用手动调分 `adjustScore(teamId,'r1',-2)` 补救 |
| 分值硬编码 2.5 | 读 `q.score_correct`（题库中 R1 全部为 2.5），兜底 2.5 |
| 走 `applyTeamScore()` 统一裁剪 | ✅ 2026-07-16 起已走统一入口（见第六章） |

**上限行为**：`applyTeamScore()` 内部执行 `actual = Math.max(0, Math.min(delta, 20 - current))`，
即**队伍分超出部分直接丢弃**。

✅ **个人分不受此裁剪**（2026-07-16 修正）：`memberScores` 加的是裁剪前的 `delta`，
因此队伍封顶后个人分仍如实累计，"最佳个人"可评。详见 6.2 第 2 条。

### 4.2 第二环节 · 团队共答（多选/多空）

**规则**：
- 5 队同时用答题板作答，限时 40 秒
- 单题 5 分
- **多选题**：正确选项集合完全一致得满分（按正确项均分）；任一错选→本题 0 分
- **多项填空**：按空均分 5 分，每空独立计分，错/漏不扣分
- 单队上限 20 分

**多选判分**：`scoreR2Multi(teamId, selected, correctOptions, totalScore = 5)`

```js
// 含任一错选 → 0 分；否则按"已选数 / 正确数"比例给分（漏选得部分分）
let hasWrong = [...selSet].some(s => !corrSet.has(s));
let delta = (!hasWrong && selSet.size > 0)
          ? (selSet.size / corrSet.size) * totalScore
          : 0;
const actual = applyTeamScore(team.id, 'r2', Math.round(delta * 100) / 100);

const fullCorrect = !hasWrong && selSet.size === corrSet.size;
const event = {
  correct: fullCorrect,
  reason:  fullCorrect ? 'correct' : (actual > 0 ? 'partial' : 'wrong'),
  delta:   actual, ...
};
```

> 实现位置：`shared.js` `scoreR2Multi()`。返回 ScoreEvent 对象（不是分数），
> 并已写入 history。

`correct` 与 `reason` 现已口径统一（2026-07-16 修正）：

| 场景 | `correct` | `reason` | 播报 |
|------|-----------|----------|------|
| 全对无错选 | `true` | `correct` | "答对，得X分" |
| 漏选（无错选） | `false` | `partial` | "部分答对，得X分" |
| 含错选 / 未选 | `false` | `wrong` | "答错，不得分" |

**多空判分**：`scoreR2FillMulti(teamId, blanks, correctBlanks, totalScore = 5)`

```js
const perBlank = totalScore / correctBlanks.length;
let correct = 0;
correctBlanks.forEach((ans, i) => {
  const given    = (blanks[i] || '').trim();
  const expected = Array.isArray(ans) ? ans : [ans];   // 每空允许多个同义词
  if (expected.some(e => e.trim() === given)) correct++;
});
const delta  = Math.round(correct * perBlank * 100) / 100;
const actual = applyTeamScore(team.id, 'r2', delta);

const fullCorrect = correct === correctBlanks.length;   // 全部空都答对
// reason 三态同多选题：correct / partial / wrong
```

> 实现位置：`shared.js` `scoreR2FillMulti()`

> ⚠️ **警告：R2 的答案比对只做 `trim()`，不做归一化。**
>
> `scoreR2FillMulti()` 是严格字符串相等比较，因此"不出卖场"与"不出卖场。"**判为不同答案**。
>
> 归一化函数 `normalizeAnswer()` 已于 2026-07-16 实现，但**只接在 R5 查重上**，
> **没有**接入 R2 判分——那会直接改变判分口径，属赛制决策，未擅自改动。
> 实际操作中由计分员肉眼比对后手工输入，风险可控。
>
> 若确认要放宽 R2 判分，把本函数的比对换成 `normalizeAnswer()` 即可（见 16.2 第 10 项）。

### 4.3 第三环节 · 擂台抢答（混合题型）

**规则**：
- 系统 TTS 读题完毕后自动喊"开始抢答"，大屏同步显示"开始抢答"字样
- **大屏显示"开始抢答"之前**按下抢答器 = 违规，系统立即扣 2 分并语音播报
- **大屏显示"开始抢答"之后**第一个按下抢答器的队伍 = 抢答成功
- 抢到后全队可沟通，指派 1 人作答，限时 15 秒
- 答对：+2 分；答错/超时：-2 分；补抢：若首抢队答错/超时，可开放一次补抢（仅一次）
- 无上下限（可为负）

**抢答状态机**（`r3.buzzState`）：

实际只有 **4 个可达状态**：`idle` → `reading` → `armed` → `locked`。

```
idle
 └─ [主持人点"出题" r3StartQuestion()] ──────→ reading
      （TTS 朗读题干 + 选项，大屏不显示"开始抢答"）
      │
      │  [任意队按抢答器 → r3EarlyBuzz()] ──→ 违规 -2 分，打断 TTS 播报违规，
      │                                        然后 resumeR3Reading() 从头重读 → 仍是 reading
      │
 reading → [TTS 队列播完（末段为"开始抢答"）] → armed
      （同一 onend 回调里启动 15 秒计时器）
      │
      │  [任意队按抢答器 → r3TryBuzz()] ────→ locked
      │
 armed → locked
      （startTimer() 重启 15 秒答题倒计时，TTS 播报"XX队抢答成功，请答题"）
      │  ✅ 2026-07-16 起答题限时自动重启；倒计时归零走 r3AutoTimeout()（见 15.4）
      │
 locked → [r3Score(true)]  ─────────────────→ idle（+score_correct，TTS 报分，自动亮答案+分数）
        → [r3Score(false)] ─────────────────→ idle（+score_wrong，TTS 报分）
        → [doR3Timeout()]  ─────────────────→ idle（主持人手动点"超时"按钮）
        → [r3AutoTimeout()] ────────────────→ 倒计时归零自动触发：扣分后
        │                                      若未用过补抢则自动转 armed，否则转 idle
        → [r3OpenSupplement()] ─────────────→ armed（补抢，仅一次，重启计时器）
```

✅ **`'supplement'` 死枚举已于 2026-07-16 清理**：枚举注释与 `index.html` 两处不可达
分支均已删除。补抢由 `r3OpenSupplement()` 实现，它把状态设回 **`'armed'`** 并置
`supplementUsed = true`；补抢是否可用靠 `supplementUsed` 布尔值判断（控制按钮禁用）。

**状态转换的守卫条件**：

| 函数 | 前置要求 | 不满足时 |
|------|---------|---------|
| `r3TryBuzz(teamId)` | `buzzState === 'armed'` | 若为 `'reading'` 则转 `r3EarlyBuzz()`；其余返回 `false` |
| `r3Score(correct)` | `selectedTeam` 且 **`selectedMember != null`** | 返回 `null`，不判分 |
| `r3OpenSupplement()` | `buzzState === 'locked'` 且 `!supplementUsed` | 返回 `false` |

> ⚠️ **注意**：`r3Score()` **要求先指定答题人**（`r3SelectMember(memberIdx)`），否则静默返回 `null`
> 什么也不做。抢答成功后 `selectedMember` 被置 `null`，必须由主持人在控制台点选队员。

**大屏三阶段显示**：

| buzzState | 大屏中央显示 | 颜色 |
|-----------|------------|------|
| `reading` | 题目内容（正常展示，无特殊提示） | 默认 |
| `armed` | 题目内容 + 顶部横幅**「开始抢答」** | 金色横幅，脉冲动画 |
| `locked` | **「XX队 抢答成功」**大字覆盖（约 1.5 秒）→ 退回题目页 | 队伍颜色 |

**违规检测实现**：

**没有 `onBuzzerKeyPress()` 分发器**。控制台的键盘监听直接调 `r3TryBuzz()`，由它内部
按状态分发：

```js
// index.html 文件末尾的 keydown 监听 —— 抢答器按键唯一入口
document.addEventListener('keydown', e => {
  if (['INPUT','TEXTAREA','SELECT'].includes(e.target.tagName)) return;
  if (curPanel !== 3) return;                      // 仅第三环节面板生效
  for (const [tid, keys] of Object.entries(state.keymap || {})) {
    if (keys.includes(e.key)) { r3TryBuzz(+tid); break; }
  }
});

// shared.js r3TryBuzz() —— 状态分发在函数内部
function r3TryBuzz(teamId, onViolationDone) {
  if (state.r3.buzzState === 'reading') return r3EarlyBuzz(teamId, onViolationDone);
  if (state.r3.buzzState !== 'armed')   return false;   // idle/locked 静默忽略
  ...
  startTimer(state.r3.timerSec * 1000, 3);   // 抢到后重启 15 秒答题倒计时
}
```

违规扣分（`shared.js` `r3EarlyBuzz()`）：

```js
function r3EarlyBuzz(teamId, onDone) {
  if (window.IS_CONTROL) stopSpeak();                      // 打断 TTS
  const delta = applyTeamScore(team.id, 'r3',
                               R3_VIOLATION_PENALTY);      // 具名常量，不读题库
  logEvent({ ..., delta, reason:'violation' }, true);      // ← skipAnnounce=true
  speakQueue([`${team.name}，抢答违规，扣两分`],
             () => resumeR3Reading(onDone));               // 播报完从头重读整题
}
```

注意 `logEvent(event, true)` 的第二参 `skipAnnounce` 置 `true`，避免 `announceScore()`
与这里的 `speakQueue` 同时发声造成双重播报。

> ⚠️ **注意：违规扣分不读题库，这是有意设计，不是遗漏。**
>
> 抢跑是**行为处罚**，与题目值几分无关——一道 5 分的题和一道 2 分的题，抢跑性质相同。
> 因此它用具名常量 `R3_VIOLATION_PENALTY`（定义在 `shared.js` 头部的配置区，
> 与 `ROUND_CAPS` 相邻），而**不读 `q.score_wrong`**（那是「答错」的分值）。
>
> 赛制若调整违规标准，**改常量一处即可**。这与下面 `r3Score()` 读题库的行为不同，
> 二者语义本就不同。

**判分函数**：`r3Score(correct)`（`shared.js` `r3Score()`）

```js
// 分值读题库，兜底 ±2（2026-07-16 修正，此前为硬编码 2 / -2）
const q     = state.questions[state.r3.currentQIdx];
const delta = correct ? (q?.score_correct ?? 2) : (q?.score_wrong ?? -2);

applyTeamScore(team.id, 'r3', delta);   // R3 无上限，applyTeamScore 原样累加（可为负）
team.memberScores[memberIdx] += delta;  // 个人分同步（可为负）
if (correct) { showAnswerOnDisplay = true; showScoresOnDisplay = true; }  // 自动亮答案
stopTimer();
```

当前题库 R3 全部为 `score_correct: 2, score_wrong: -2`，所以实际结果不变；
但改用题库配置后，将来引入异分值题目不会再静默判错。

**与设计稿的差异**：设计稿的 `scoreR3(teamId, memberIdx, correct, isViolation)` 不存在；
违规与判分是两个独立函数（`r3EarlyBuzz()` / `r3Score()`），且都从 `state.r3` 读上下文
而非传参。

### 4.4 第四环节 · 识图找茬

**规则**：
- 每队 1 张图（共 5 图，提前抽签分配）
- 每图预设 10 个找茬点，60 秒内口述
- 找对 1 处 +1 分，最高 10 分，不扣分
- 评委可临时认定预设之外的合理违规点（仍封顶 10 分）
- 超时未表述完的不计分；重复描述不重复计分

**评委端操作**：
- 10 个点位复选框（勾选=有效）
- 额外可输入自定义找茬点（评委认定后添加）
- 实时汇总分数（min(已勾选总数, 10)）

**判分函数**：`scoreR4(teamIdx)`（`shared.js` `scoreR4()`）

注意参数是 **`teamIdx`（出场顺序下标）**，不是 `teamId`；勾选结果与额外点从
`state.r4` 读取，不传参：

```js
function scoreR4(teamIdx) {
  const team  = getTeamByIdx(teamIdx);            // 经 draw.teamOrder 转换
  const found = Object.values(state.r4.spotJudge).filter(Boolean).length
                + state.r4.extraSpots.length;
  // 先按单图上限 10 收口，再走统一入口做队伍累计上限裁剪
  const actual = applyTeamScore(team.id, 'r4', Math.min(found, ROUND_CAPS.r4));
  logEvent({ round:4, ..., delta:actual, reason:'spot', foundCount:found });
}
```

`foundCount` 记录裁剪**前**的真实找出点数，`delta` 是裁剪后实际给分，便于赛后复盘。

**相关函数**：
- `r4JudgeSpot(spotKey, found)` —— 勾选单个点位。`spotKey` 是题库
  `spots[].key` 的字符串（如 `"a1"`），不是数字下标。
- `r4AddExtraSpot(desc)` —— 追加评委认定的额外点，存 `{desc, ts}` 对象。
- `getR4ImageSrc(imageKey)` —— `imageKey`（如 `'图A'`）→ 图片路径。先查
  `state.r4.imageFiles`，查不到则回落约定路径 `images/{imageKey}.png`。
  大屏据此显示场景图，文件缺失时显示红色占位提示（见 9.x）。

✅ **勾选数已设上限**（2026-07-16 实现）：`r4JudgeSpot(key, true)` 与
`r4AddExtraSpot(desc)` 在 `r4FoundCount() >= R4_MAX_SPOTS`（10）时**返回 `false` 并拒绝**。
控制台 UI 同步把未勾选项置灰、计数变黄显示"（已满 10 处）"，并弹提示。

两条重要的设计约束：

1. **只拦"新增"，取消勾选恒放行**——否则满 10 后评委无法纠错，会把误勾锁死。
2. **额外认定点可删除**（`r4RemoveExtraSpot(idx)`，UI 上每个点后有 `×`），
   同样是为了满额后仍能改判。

> ⚠️ **注意**：此前 `index.html` 的 `r4ToggleSpot()` **直接改 `state.r4.spotJudge`、绕过
> `r4JudgeSpot()`**，导致业务规则形同虚设。现已改为必须经 `r4JudgeSpot()`。
> **新增任何 R4 勾选入口时都必须走该函数**，不要直接写 state。

### 4.5 第五环节 · 服务飞花令

**规则**：
- 共 3 个令题，每令题各队按顺序轮流接龙
- 每次 8~10 秒内说出 1 条有效内容（全队可商议）
- 有效（不重复、不违规）：+1 分
- 出局（重复/说错/超时）：退出本令题后续轮次，当次不扣分
- 本令题最后存活 1 队 = 擂主，额外 +3 分
- 若最终并列，以新令题加赛决胜

**状态机**（单令题）：
```
r5StartTheme(idx) → activeTeams = 全部 5 队，currentTurnIdx = 0
   │
   ├─ r5ValidAnswer(teamId, answer) → +1 分，记入 usedAnswers → _r5NextTurn()
   │
   └─ r5Eliminate(teamId) → 从 activeTeams 移除
           │
           ├─ 剩余 > 1 队 → _r5NextTurn()
           └─ 剩余 == 1 队 → 自动 r5SetWinner(最后一队) → +3 分
```

**判分函数**：

```js
// shared.js r5ValidAnswer() —— 有效答案，注意有第二参 answer
function r5ValidAnswer(teamId, answer) {
  state.r5.usedAnswers.push({ teamId, answer, ts: Date.now() });
  applyTeamScore(team.id, 'r5', 1);   // R5 无上限，原样累加
  logEvent({ round:5, ..., delta:1, reason:'flower_valid', answer });
  _r5NextTurn();            // 自动推进到下一队
}

// shared.js r5Eliminate() —— 出局
function r5Eliminate(teamId) {
  state.r5.activeTeams = state.r5.activeTeams.filter(id => id !== teamId);
  logEvent({ ..., delta:0, reason:'wrong', eliminated:true }, true);  // skipAnnounce
  if (window.IS_CONTROL) speak(`${team.name}出局`);
  if (state.r5.activeTeams.length === 1) r5SetWinner(state.r5.activeTeams[0]);  // 自动决出擂主
  else _r5NextTurn();
}

// shared.js r5SetWinner() —— 擂主 +3
function r5SetWinner(teamId) {
  applyTeamScore(team.id, 'r5', 3);
  state.r5.themeWinners.push({ themeIdx: state.r5.currentThemeIdx, teamId });
  state.showScoresOnDisplay = true;      // 自动亮分
}
```

**轮次推进**（`shared.js` `_r5NextTurn()`）：从 `currentTurnIdx + 1` 起在
`r5.teamOrder` 上环形查找下一支仍在 `activeTeams` 中的队伍，带 `guard` 计数防死循环；
落位后**自动启动 10 秒计时器**。倒计时归零由 `onTimerExpired(5)` 自动判当前队出局
（见 15.4）。

✅ **答案去重已实现**（2026-07-16）：`r5ValidAnswer(teamId, answer, opts)` 在计分前
调 `r5FindDuplicate()` 查重。

**返回值有两种形态，调用方必须区分**：

| 情况 | 返回 | 副作用 |
|------|------|--------|
| 未重复 | ScoreEvent（`delta:1`） | 计分 + 写 `usedAnswers` + 推进轮次 |
| **命中重复** | `{ duplicate:true, prev:{teamId,teamName,answer,ts} }` | **无任何副作用**——不计分、不写入、不推进轮次 |
| 队伍不存在 | `null` | 无 |

命中重复时控制台弹确认框，展示本次答案、撞车的已有答案、以及是哪队何时说的，
让评委二选一：判重复出局，或裁定"不算重复"并以 `{force:true}` 强制通过。

**比对经 `normalizeAnswer()` 归一化**（这就是 16.2 第 10 项要的 `normalize()`）：
去首尾空白 → 全角转半角 → 去中英文标点 → 折叠内部空白 → 转小写。
因此"微笑服务"、"微笑服务。"、" 微笑　服务 " 会被判为同一答案。

> ⚠️ **空答案与 `R5_BLANK_ANSWER`（`（未录入）`）不参与查重**。它们不是真实答案，
> 若参与则两队都留空会被判成互相重复。

> ⚠️ **注意**：`normalizeAnswer()` 目前**只用于 R5 查重**。R2 的 `scoreR2FillMulti()`
> 仍是 `trim()` 后严格相等——改动它会直接影响判分口径，属赛制决策，未擅自改
> （见 16.2 第 10 项）。

> ⚠️ **`r5.timerSec` 被 `initR5()` 强制重置为 10**，即使用户在设置里改过也会被覆盖。
> 赛制说 8~10 秒，若要用 8 秒需在 `initR5()` 之后再改。

#### 并列加赛（赛制：若最终并列，以新令题加赛决胜）

✅ **已实现**（2026-07-16）。三个函数：

| 函数 | 说明 |
|------|------|
| `hasTie(rank = 1)` | 检测第 `rank` 名的并列，返回并列队伍数组；**不足 2 队返回空数组** |
| `getTieGroups()` | 返回全部并列组 `[{total, teams[]}]`（含非榜首的并列，供赛后核对名次） |
| `r5StartTiebreak(themeIdx, rank = 1)` | 发起加赛；无并列时返回 `false` 不启动 |

`r5StartTiebreak()` 的行为：

- **只有并列的队伍参加**——`r5.teamOrder` 按 `draw.teamOrder` 过滤，**保持原出场顺序**
- 置 `r5.isTiebreak = true`，控制台令题标签与大屏令题区都显示「加赛」标记
- 其余流程完全复用常规令题：`r5ValidAnswer()` / `r5Eliminate()` / 仅剩 1 队自动成擂主

控制台 P5 的 `[并列检测 / 加赛]` 按钮（`doCheckTie()`）：

1. 无并列 → 提示"冠军无并列，无需加赛"，并附带列出非榜首的并列组
2. 有并列 → 自动挑一道**尚未产生过擂主**的令题，确认后发起加赛
3. 令题已用完 → 提示需人工裁定或补充令题

> ⚠️ **注意**：加赛后调 `r5StartTheme(idx)` 会把 `r5.teamOrder` **还原为全部 5 队**
> 并清除 `isTiebreak`。这是有意的——加赛只作用于当前令题。

### 4.6 总分计算

```js
function teamTotal(team) {
  return Object.values(team.scores).reduce((a, b) => a + (b || 0), 0);
}
```

总分 = r1 + r2 + r3 + r4 + r5（r3 可为负），无封顶，最高者胜。

### 6.1 实际实现：applyTeamScore() 统一入口（2026-07-16 重构）

所有环节判分统一走 **`applyTeamScore(teamId, roundKey, delta)`**（`shared.js`）：

```js
function applyTeamScore(teamId, roundKey, delta) {
  const team = getTeam(teamId);
  if (!team) return 0;
  const cap     = ROUND_CAPS[roundKey];
  const current = team.scores[roundKey] || 0;
  let actual;
  if (cap == null || delta < 0) {
    actual = delta;                                        // 无上限环节 / 扣分：原样
  } else {
    actual = Math.max(0, Math.min(delta, cap - current));  // 加分：裁剪到上限
  }
  team.scores[roundKey] = current + actual;
  return actual;   // 实际生效的分差；不 save()，由调用方负责
}
```

| 情形 | 行为 |
|------|------|
| 无上限环节（r3/r5，`cap == null`） | 原样累加，可正可负 |
| 有上限环节**加分** | `clamp(delta, 0, cap - current)`，超出部分丢弃；已封顶时不会倒扣 |
| 有上限环节**扣分** | **原样通过，不受上限约束** |

> ⚠️ **注意：扣分为什么不裁剪？**（2026-07-16 修正）
>
> 原实现是 `Math.max(0, Math.min(delta, cap - current))`，对**负** `delta` 会得到
> `Math.max(0, 负数) = 0`——**扣分被静默吞掉**。
>
> 此前无人踩到，是因为当时没有任何"有上限环节的扣分"场景：r1/r2/r4 只加不减，
> r3/r5 又都是无上限。直到实现 **R1 违规扣 2 分**（16.1 第 5 项）才暴露。
>
> 语义上，处罚就是处罚，与队伍当前是否封顶无关：一支已拿满 20 分的队伍违规，
> 必须真的扣到 18 分。因此扣分不套上限。

- 调用点：`scoreR1()`、`scoreR2Multi()`、`scoreR2FillMulti()`、`scoreR4()`、
  `r3Score()`、`r3EarlyBuzz()`、`r3Timeout()`、`r5ValidAnswer()`、`r5SetWinner()`，
  以及 `index.html` 的 `doR2Score()`

### 6.2 已知问题

1. **`adjustScore()` 绕过所有上限**：手动调分直接
   `team.scores[round] += delta`，可以把 r1 调到 20 分以上。这是有意的设计（评委裁定
   需要凌驾于规则之上），但操作时需注意。

2. ~~**个人分与队伍分的裁剪耦合**~~ —— ✅ **已于 2026-07-16 解耦**。
   `scoreR1()` 现在队伍分加裁剪后的 `actual`、个人分加裁剪**前**的 `delta`。

   **为什么必须解耦**：R1 满分 20 = 4 人 × 2 题 × 2.5 分，**全队全对必然正好触顶**。
   若个人分跟着裁剪，恰恰是表现最好的队伍评不出「最佳个人」——这不是边缘情况，
   而是必然发生。

   代价：封顶后 `sum(memberScores) > scores.r1`，二者不再恒等。ScoreEvent 因此
   新增 `memberDelta`（个人实际入账）与 `capped`（是否触顶）两个字段以便复盘，
   `delta` 仍表示**队伍**实际入账。控制台封顶时会播报"本环节已达上限，本题不再加分"，
   避免台下误以为漏计。

### 7.3 新增题目时的检查清单

1. 改 `raw_qbank.txt`，不改 `questions.json`。
2. 跑 `python parse_qbank.py`，确认 `validate_out.txt` 中 `Errors: 0`。
3. 题干字段名是 `stem`；分值是 `score_correct`（下划线）。
4. `options` 不要写 "A. " 前缀。
5. R3 的题**必须带 `score_wrong`**（现行题库全部为 `-2`）。`r3Score()` 会读它，
   缺失时兜底 -2。
6. R3 已支持非 ±2 的分值（2026-07-16 修复），`score_correct` / `score_wrong` 会被
   如实读取。但请确认赛制文件与主持人口径同步，避免现场争议。
7. `spot` 题的 `score_correct` **不会被读取**，`scoreR4()` 固定每处 +1（见 16.3 第 21 项）。

### 10.2 跨窗口协作的注意事项

1. **只有控制台发声、启动计时**。`shared.js` 中大量逻辑用 `if (window.IS_CONTROL)`
   包裹（TTS、`startTimer` 调用等）。大屏加载同一份 `shared.js` 但 `IS_CONTROL`
   为 `undefined`，因此静默跳过。**在 `shared.js` 里新增会产生副作用的逻辑时，
   务必考虑它会在两个窗口各跑一次。**

2. **倒计时不同步数值，只同步起点**。`state.timer` 存的是 `startedAt` 时间戳，
   两个页面各自用 `getRemainingMs()` 本地计算剩余时间（`requestAnimationFrame` 轮询）。
   这样避免了每秒广播造成的抖动与漂移。**代价是依赖两台机器时钟一致**——同一台电脑
   开两个窗口无此问题。

3. **`stopTimer()` 不调 `save()`**（`shared.js` `stopTimer()`），它只改内存中的
   `state.timer.state`。调用方必须自己 `save()`，否则大屏不会同步。
   如 `r3Score()` 是先 `stopTimer()` 再 `save()`。
   对比之下 `startTimer()` / `pauseTimer()` / `resumeTimer()` / `resetTimer()`
   **都自带 `save()`**，这个不一致容易踩。

   ⚠️ `r3TryBuzz()` 现在结尾调的是 **`startTimer()`**（重启 15 秒答题倒计时），
   由它内部完成 `save()`，因此函数里看不到显式的 `save()` 调用——不是漏了。

### 12.1 设计背景与目标

比赛中主持人仅负责串场与流程推进，**题目宣读、选项朗读、环节过渡语、倒计时提示、
抢答判定播报**等全部由系统自动完成，无需外接设备。

语音有两条链路（详见 **12.11**）：

| 链路 | 说明 |
|------|------|
| **本地 TTS 服务**（推荐） | `tts_server/`，神经语音，音色自然、跨设备一致。**可选组件** |
| **浏览器原生** | Web Speech API / SpeechSynthesis，零依赖但音色机械、各设备不一 |

默认 `tts.engine = 'auto'`：服务可用就用服务，否则回退原生。
**不启动服务时，本章描述的全部话术与流程照常工作**，只是音色变回系统语音。

**目标**：
- 解放主持人，减少人为宣读失误
- 语速、音色可配置，适配不同场地音响
- 所有朗读内容与系统操作事件绑定，触发准确、时机合理
- 控制台保留手动打断 / 重读 / 跳过按钮，应对突发情况

### 12.2 技术方案

**使用 Web Speech API（SpeechSynthesis）**，无需任何第三方库或后端接口。

```js
// 核心对象
const synth = window.speechSynthesis;

// 创建话语
function speak(text, options = {}) {
  synth.cancel();  // 打断当前正在播放的内容
  const utt = new SpeechSynthesisUtterance(text);
  utt.lang   = options.lang  ?? state.tts.lang;    // 默认 'zh-CN'
  utt.rate   = options.rate  ?? state.tts.rate;    // 默认 1.0（0.5~2.0）
  utt.pitch  = options.pitch ?? state.tts.pitch;   // 默认 1.0
  utt.volume = options.volume ?? state.tts.volume; // 默认 1.0
  utt.voice  = getTTSVoice();
  utt.onend  = options.onend ?? null;
  synth.speak(utt);
  return utt;
}

// 队列式朗读（多段依次播放）
function speakQueue(segments, onAllDone) {
  if (!segments.length) { onAllDone?.(); return; }
  const [first, ...rest] = segments;
  speak(first, { onend: () => speakQueue(rest, onAllDone) });
}
```

**兼容性**：Chrome 33+（含 Chromium 内核）原生支持，无需 polyfill。中文语音需操作系统安装有中文 TTS 引擎（Windows 10/11 默认已有「微软云希」/「微软晓晓」等）。

```js
tts: {

  lang: 'zh-CN',          // 语言代码

#### 12.5.2 分值转中文朗读

TTS 直接朗读"2.5"会读成"二点五"，"10"会读成"十"，需要统一转换为自然中文，避免机械感：

```js
/**
 * 将分值数字转换为适合 TTS 朗读的中文字符串
 * 1 → "一"，2 → "两"，2.5 → "两点五"，10 → "十"
 */
function scoreToSpeech(n) {
  if (n === 0) return '零';
  const abs = Math.abs(n);
  // 整数部分
  const intPart = Math.floor(abs);
  const decPart = Math.round((abs - intPart) * 10);  // 最多一位小数
  const intMap = { 1:'一', 2:'两', 3:'三', 4:'四', 5:'五',
                   6:'六', 7:'七', 8:'八', 9:'九', 10:'十',
                   20:'二十' };
  const intStr = intMap[intPart] ?? String(intPart);
  if (decPart === 0) return intStr;
  const decMap = { 1:'一', 2:'二', 3:'三', 4:'四', 5:'五' };
  return `${intStr}点${decMap[decPart] ?? decPart}`;
}
// 示例：scoreToSpeech(2.5) → "两点五"
//        scoreToSpeech(10)  → "十"
//        scoreToSpeech(3)   → "三"
```

#### 12.5.3 报分函数

```js
/**
 * 判分完成后调用，自动生成并播报得分语音
 * @param {ScoreEvent} event  logEvent 产生的判分事件
 */
function announceScore(event) {
  if (!state.tts.enabled) return;

  const name = event.teamName;
  const delta = event.delta;
  let text = '';

  switch (event.reason) {
    case 'correct':
      text = `${name}，答对，得${scoreToSpeech(delta)}分。`;
      break;
    case 'wrong':
      text = delta < 0
        ? `${name}，答错，扣${scoreToSpeech(-delta)}分。`
        : `${name}，答错，本题零分。`;
      break;
    case 'timeout':
      text = delta < 0
        ? `${name}，超时，扣${scoreToSpeech(-delta)}分。`
        : `${name}，超时，本题零分。`;
      break;
    case 'violation':
      text = `${name}，违规，扣${scoreToSpeech(-delta)}分。`;
      break;
    case 'spot':
      text = `${name}，找到${scoreToSpeech(delta)}处，得${scoreToSpeech(delta)}分。`;
      break;
    case 'flower_valid':
      text = `${name}，有效，得一分。`;
      break;
    case 'flower_winner':
      text = `${name}，擂主，额外得三分。`;
      break;
    case 'adjust':
      text = delta >= 0
        ? `${name}，手动加${scoreToSpeech(delta)}分。`
        : `${name}，手动扣${scoreToSpeech(-delta)}分。`;
      break;
  }

  if (text) speak(text, { priority: 'P2' });  // P2 优先级，打断读题但不打断违规播报
}
```

`announceScore(event)` 在每次 `logEvent()` 写入 history 后自动调用，无需手动触发。

### 12.6 朗读触发机制

朗读统一由控制台（`index.html`）驱动 ✅，但**不是通过 onChange 钩子自动检测**，
而是在各操作函数里显式调用。

> ⚠️ 设计稿的 `onQuestionChanged()` / `checkAndTriggerTTS()` **都不存在**，
> `onChange(() => checkAndTriggerTTS())` 这个监听器也不存在。
> 若照此实现会与现有流程冲突（重复朗读）。

**真实的触发点**：

| 场景 | 触发函数 | 位置 |
|------|---------|------|
| R1 翻牌后自动读题 | `doFlipCard()` 的 `setTimeout` 回调 → `if (state.tts.autoRead) r1ReadQ()` | `index.html` `doFlipCard()` |
| R2 翻牌后自动读题 | 同上 → `r2ReadQ()` | `index.html` `doFlipCard()` |
| R1/R2 手动读题 | `r1ReadQ()` / `r2ReadQ()` 按钮 | `index.html` |
| R3 出题读题 | `r3StartQuestion()` **自身发起**（在 `shared.js` 内） | `shared.js` `r3StartQuestion()` |
| R3 违规后重读 | `resumeR3Reading()` | `shared.js` `resumeR3Reading()` |
| R4 翻牌 | **不朗读**，直接 `startTimer()` | `index.html` `doFlipCard()` |
| 超时播报 | `onTimerExpired()` / `r3AutoTimeout()` | `index.html`，见 15.4 |

```js
// index.html —— 翻牌 1.5 秒后（等大屏播完翻牌动画）确认并读题
setTimeout(() => {
  confirmFlip(id);
  if (round === 1) {
    if (state.tts.autoRead) r1ReadQ();
    else { startTimer(state.r1.timerSec * 1000, 1); cdSpoken.clear(); }
  } else if (round === 2) {
    if (state.tts.autoRead) r2ReadQ();
    else { startTimer(state.r2.timerSec * 1000, 2); cdSpoken.clear(); }
  } else if (round === 4) {
    startTimer(state.r4.timerSec * 1000, 4); cdSpoken.clear();
  }
  render();
}, 1500);

// shared.js —— R3 出题时自行读题，读完自动 armed 并启动计时
if (window.IS_CONTROL) {
  const segs = [...buildQuestionSegments(q), '开始抢答'];
  speakQueue(segs, () => {
    state.r3.buzzState = 'armed';
    save();
    onArmed?.();
    startTimer(state.r3.timerSec * 1000, 3);
  });
}
```

> ⚠️ **警告：`autoRead` 的 `else` 分支必须自己启动计时器，极易漏改。**
>
> 计时器有**两条互斥的启动路径**，取决于 `state.tts.autoRead`：
>
> | `autoRead` | 读题 | 谁启动计时 |
> |:---|:---|:---|
> | `true` | `r1ReadQ()` / `r2ReadQ()` | **TTS 的 `onend` 回调**（读完才起计时，选手不会被读题时间吃掉） |
> | `false` | 跳过 | **`else` 分支必须手动 `startTimer(...)`** |
>
> 两条分支都要负责起计时，是重复逻辑。**新增环节或改动此处时，若只写了
> `if (autoRead) xxReadQ();` 而忘了 `else`，计时器将永远不启动**——
> 且现场表现为"倒计时不动"，而非报错，很难当场定位。
>
> 第四环节没有这个分支：它从不朗读，无条件 `startTimer()`。

**触发优先级**（高优先级打断低优先级）：

| 优先级 | 场景 |
|--------|------|
| P1（最高） | 违规判定、抢答成功提示 |
| P2 | 语音报分（答对/答错/得分/扣分，见 12.5） |
| P3 | 倒计时提示 |
| P4 | 题目正文朗读 |
| P5（最低） | 环节过渡语 |

高优先级调用 `synth.cancel()` 打断当前朗读，低优先级排队等待。

### 12.7 计时器与 TTS 联动

```
[主持人点击"出题/开始计时"]
    ↓
系统开始读题（speakQueue 读题目 + 选项）
    ↓
读题完毕（onend 回调）→ 自动启动倒计时
    ↓
计时器运行，到达 countdownAt 节点时插入播报
    （如剩余10秒、5秒、3/2/1秒）
    ↓
倒计时归零 → 播报"时间到" → 自动判超时
```

**第三环节完整朗读与抢答判定流程**：

```js
/**
 * 主持人点击"出题"后调用
 * 1. 进入 reading 状态（此时按抢答器=违规）
 * 2. TTS 读完题目 + 选项后，自动说"开始抢答"
 * 3. "开始抢答"说完的 onend 回调里切换到 armed 状态，大屏同步显示横幅，并启动抢答计时
 *
 * ⚠️ 第一参是 qIdx（questions[] 的索引），不是题目对象
 */
function r3StartQuestion(qIdx, onArmed) {
  const q = state.questions[qIdx];
  if (!q) return false;

  // 进入 reading 状态，记录当前题目文本（违规后重读用）
  state.r3.currentQIdx     = qIdx;
  state.r3.buzzState       = 'reading';
  state.r3.currentReadText = q.stem || '';
  save();

  // 仅控制台朗读；大屏 IS_CONTROL 为 undefined，直接走 else 分支
  if (window.IS_CONTROL) {
    const segs = [...buildQuestionSegments(q), '开始抢答'];
    speakQueue(segs, () => {
      state.r3.buzzState = 'armed';
      save();                                   // 大屏据此显示"开始抢答"横幅
      onArmed?.();
      startTimer(state.r3.timerSec * 1000, 3);  // 抢答窗口计时
    });
  } else {
    onArmed?.();
  }
  return true;
}

/**
 * 违规抢答：打断 TTS → 播报违规 → 重读题目 → 恢复 reading 状态
 */
function r3EarlyBuzz(teamId, onDone) {          // shared.js（实际签名有第二参）
  if (window.IS_CONTROL) stopSpeak();          // 打断当前读题（非 synth.cancel()）
  const team = getTeam(teamId);                // 非 getTeamName()，该函数不存在
  applyTeamScore(team.id, 'r3', -2);           // 2026-07-16 起走统一入口
  logEvent({ round:3, teamId, teamName:team.name, correct:false,
             delta:-2, reason:'violation', ts:Date.now() },
           true);                              // ← skipAnnounce=true，避免与下方 speakQueue 双重播报

  // 播报违规后交给 resumeR3Reading()：它会把 buzzState 设回 'reading'，
  // 从头重读整题 + "开始抢答"，播完再进 'armed' 并启动计时
  speakQueue(
    [`${team.name}，抢答违规，扣两分`],
    () => resumeR3Reading(onDone)
  );
}

/**
 * 合法抢答成功（armed 状态下第一个按键的队伍）
 * ⚠️ 实际是抢答总入口：reading 状态下调用会转为 r3EarlyBuzz()
 */
function r3TryBuzz(teamId, onViolationDone) {
  if (state.r3.buzzState === 'reading') return r3EarlyBuzz(teamId, onViolationDone);
  if (state.r3.buzzState !== 'armed') return false;
  const team = getTeam(teamId);
  if (!team) return false;

  state.r3.buzzState      = 'locked';
  state.r3.buzzedTeam     = teamId;
  state.r3.selectedTeam   = teamId;
  state.r3.selectedMember = null;                // 必须由主持人点选队员，见 4.3
  state.r3.buzzPulse      = (state.r3.buzzPulse || 0) + 1;
  state.r3.lastBuzzTeam   = teamId;

  // 抢答窗口结束，重启【答题】15 秒倒计时（2026-07-16 补齐）
  // startTimer() 内部自带 save()，故此处无显式 save——不是漏了
  startTimer(state.r3.timerSec * 1000, 3);

  if (window.IS_CONTROL) speak(`${team.name}抢答成功，请答题`);
  return true;
}
```

> ⚠️ **警告：设计稿的 `startTimer(15, 3, true)` 是错的，切勿照抄。**
>
> | 错误点 | 说明 |
> |:---|:---|
> | 第一参单位 | 是**毫秒**不是秒。`15` 会得到 15 毫秒，应为 `state.r3.timerSec * 1000` |
> | 第三参 | **不存在**。实际签名是 `startTimer(durationMs, round = null)`，只有两个参数 |
> | `autoExpire` | 该字段已于 2026-07-16 从 `state.timer` 删除，无任何代码读它 |
>
> 正确写法：`startTimer(state.r3.timerSec * 1000, 3)`。详见 15.3。

### 12.11 双引擎架构（本地 TTS 服务 + 原生兜底）

浏览器原生 Web Speech 的音色取决于操作系统自带语音，音质较差、机械感强，且不同设备
表现不一致、不可控。v3.1 起引入**可选的本地 TTS 服务**（`tts_server/`），将语音合成
从浏览器移交至可控的本地服务，以解决音质与跨设备一致性问题。

```
speak(text) / speakQueue(segs)
    │
    ├─ _useServer() == true  ──→ POST {serverUrl}/api/speak ──→ <audio> 播放
    │      │
    │      └─ 任何失败（服务未启动/断线/超时/音频损坏）
    │             └──→ 自动回退 ↓（并置 _serverOk=false，后续直接走原生）
    │
    └─ _useServer() == false ──→ speechSynthesis（浏览器原生）
```

**关键设计：服务是可选的。** 不启动 `tts_server`，系统行为与 v3.0 完全一致。
控制台启动时探测一次，之后每 30 秒复查（`index.html` 末尾的 `setInterval`），
服务中途启停都能自动跟上。**比赛不会因为语音服务挂掉而中断。**

#### 引擎选择（`state.tts.engine`）

| 值 | 行为 |
|----|------|
| `'auto'`（默认） | 健康检查通过就用服务，否则原生 |
| `'server'` | 强制服务；服务挂了仍会回退原生，但不再自动重连 |
| `'native'` | 强制原生，完全跳过健康检查 |

#### 相关状态字段

| 字段 | 说明 |
|------|------|
| `tts.engine` | `auto`/`server`/`native` |
| `tts.serverUrl` | 默认 `http://127.0.0.1:5231` |
| `tts.serverVoice` | 服务端音色名，空=服务端默认 |
| `tts.voiceName` | **仅原生引擎**使用的系统音色 |

#### 相关函数（`shared.js`）

| 函数 | 说明 |
|------|------|
| `ttsCheckServer()` | 探测 `/api/health`，写 `_serverOk` / `_serverInfo`，返回 info |
| `ttsServerStatus()` | `{ok, info, using:'server'|'native'}`，供 UI 显示 |
| `ttsPrewarm()` | 把固定话术 + 各队名播报送去服务端并发预合成 |
| `_prefetchAll(segs)` | 队列播放前并行发起全部合成请求 |

#### 打断隔离（`_speakSeq`）

服务链路是异步的，必须防止「过期的音频在被打断后才返回并播放」。
`_cancelAll()` 每次自增 `_speakSeq`，异步回调回来先比对序号，对不上就丢弃：

```js
const url = await _fetchAudio(text);
if (seq !== _speakSeq) return;   // 已被新播报打断，丢弃
```

已实测：先 `speak(甲)`、50ms 后 `speak(乙)`，甲的响应回来时被正确丢弃，只播乙。

#### 延迟与预热（**赛前必做**）

单条合成约 **3 秒**（edge 引擎，含网络往返），命中缓存约 **15 毫秒**。
倒计时"3、2、1"不预热会明显拖拍。

| 场景 | 未预热 | 已预热 |
|------|--------|--------|
| 单条固定话术 | ≈3400ms | ≈15ms |
| 6 段读题（题干+4选项+开始抢答） | 首段 3.3s，全程 9.2s | 首段 3.3s（题目动态，无法预热） |

> ⚠️ **注意**：读题必须走 `speakQueue()`，它会 `_prefetchAll()` 并行发起全部合成。
> 若改成逐段 `speak()` 串行等待，6 段要 20.5 秒——现场无法接受。

> ⚠️ **改过队名后要重新点一次预热**（话术里含队名）。

#### 三层去重

`_fetchAudio()` 依次检查：客户端 `_audioCache`（objectURL）→ `_inflight`
（同文本并发请求共享同一 Promise）→ 发新请求。服务端另有一层文件缓存。

详细部署与选型见 [`tts_server/README.md`](tts_server/README.md)。

### 12.12 已知限制与应对

| 限制 | 应对方案 |
|------|---------|
| Chrome 长时间空置后 SpeechSynthesis 可能静默 | ✅ 已处理：`speak()` / `_nativeSpeak()` 每次朗读前 `cancel()` 清队列，`speak()` 后调 `resume()` 兜底解除 paused 静默 |
| 原生引擎音色机械、跨设备不一致 | ✅ 已处理：启用 `tts_server` 走神经语音（见 12.11） |
| 本地 TTS 服务未启动 / 中途挂掉 | ✅ 已处理：自动回退原生，30 秒复查自动重连 |
| 服务链路首句约 3 秒延迟 | 赛前点「预热」；题目正文无法预热，靠 `_prefetchAll()` 并行收敛 |
| 某些数字/专业词汇发音不准 | 在模板中用中文全拼替代（如「加两分」而非「+2分」）；或用 `tts.scripts` 覆盖话术（12.10） |
| 多段队列在快速操作时可能错乱 | ✅ 已处理：`_speakSeq` 打断隔离 + 每次 `_cancelAll()` |
| 浏览器自动播放限制 | 页面上先点任意按钮取得用户手势 |
| 大屏页面也加载了 shared.js | display.html 不调用任何 TTS 函数（`isTTSAvailable()` 要求 `window.IS_CONTROL`） |
| `file://` 打开时改了 shared.js 不生效 | Chrome 对 file:// 子资源缓存极顽固，硬刷新无效。改用 `python -m http.server 8080` 调试 |

### 13.2 抽签流程

#### 队伍出场顺序抽签

```

### 13.3 状态设计

// 在全局 state 中
draw: {
  // 队伍出场顺序（5 个 team.id，按出场先后排列）
  // 示例：[3, 1, 5, 2, 4] 表示绿队第1出场，红队第2……
  teamOrder: [],

  // 识图找茬图片分配（key = team.id，value = 图片序号 0~4）
  // 示例：{ 1: 2, 2: 0, 3: 4, 4: 1, 5: 3 }
  r4ImageMap: {},

  // 锁定标志（true 后不可重抽）
  orderLocked: Boolean,
  imageLocked: Boolean,

  // 抽签流水记录
  log: [
    // { type: 'order'|'image'|'force_override', result: any, ts: Number, operator: String }
  ],
}

### 13.4 核心函数

```js
/**
 * 随机打乱 5 队出场顺序（复用 shared.js 的 fisherYates() 工具函数）
 * 返回 false 表示已锁定、未执行
 */
function drawTeamOrder() {
  if (state.draw.orderLocked) return false;
  const ids = state.teams.map(t => t.id);
  state.draw.teamOrder = fisherYates(ids);
  state.draw.log.push({ type:'draw_order', result:[...state.draw.teamOrder], ts: Date.now() });
  save();
  return true;
}

/**
 * 将图题分配给 5 支队伍
 * ⚠️ 必须传入 imageKeys（如 ['图A','图B','图C','图D','图E']），函数不自己去题库取
 * ⚠️ 按 draw.teamOrder 的顺序分配，因此必须先抽出场顺序
 */
function drawR4Images(imageKeys) {
  if (state.draw.imageLocked) return false;
  const shuffled = fisherYates(imageKeys).slice(0, 5);
  state.draw.r4ImageMap = {};
  state.draw.teamOrder.forEach((teamId, i) => {
    state.draw.r4ImageMap[teamId] = shuffled[i];            // 值是 imageKey 字符串
  });
  state.draw.log.push({ type:'draw_images', result:{...state.draw.r4ImageMap}, ts: Date.now() });
  save();
  return true;
}

/** 锁定出场顺序（确认后 drawTeamOrder 失效） */
function lockDrawOrder() {
  if (!state.draw.teamOrder.length) return false;
  state.draw.orderLocked = true;
  state.draw.log.push({ type:'lock_order', ts: Date.now() });
  save();
  return true;
}

/** 锁定图题分配 */
function lockDrawImages() {
  if (!Object.keys(state.draw.r4ImageMap).length) return false;
  state.draw.imageLocked = true;
  state.draw.log.push({ type:'lock_images', ts: Date.now() });
  save();
  return true;
}

/** 清空抽签并解锁（函数名是 resetDraw，不是 unlockDraw） */
function resetDraw() {
  state.draw = { teamOrder: [], r4ImageMap: {}, orderLocked: false, imageLocked: false, log: [] };
  save();
}
```

#### ⚠️ 与设计稿的差异

| 设计稿 | 实际 |
|-------|------|
| `drawR4Images()` 无参，内部生成 `[0,1,2,3,4]` | **`drawR4Images(imageKeys)` 必须传数组**，值是 `"图A"` 这类字符串 |
| `r4ImageMap` 值为数字 0~4 | 值为 **imageKey 字符串**（与题库 `q.imageKey` 对应） |
| 按 `state.teams` 顺序分配 | 按 **`draw.teamOrder`** 分配（须先抽顺序，否则 map 为空） |
| `unlockDraw()` | **`resetDraw()`** |
| `setTeamOrderManual(orderArr)` | ✅ **已实现**（2026-07-20）。锁定前的正常录入路径，不需操作人备注；校验 orderArr 必须是全部队伍的排列；锁定后拒绝。控制台 P0 提供拖拽排序 UI |
| `setR4ImageManual(teamId, imgIdx)` | ❌ **不存在**。手动指定图片未实现 |
| log `type: 'order'` / `'image'` | **`'draw_order'` / `'draw_images'`**；另有 `'lock_order'` / `'lock_images'` / `'force_override'` |
| `clearAllScores()` 时自动解锁 | ❌ **不会**。`clearAllScores()` 不碰 `state.draw`，须单独调 `resetDraw()` |

#### `draw.log` 的实际事件类型

| `type` | 附带字段 | 触发 |
|--------|---------|------|
| `draw_order` | `result: teamId[]` | `drawTeamOrder()` |
| `draw_images` | `result: {teamId: imageKey}` | `drawR4Images()` |
| `lock_order` | — | `lockDrawOrder()` |
| `lock_images` | — | `lockDrawImages()` |
| `force_override` | `prev`, `result`, `operator` | `forceOverrideTeamOrder()` |

### 13.7 防错与约束

| 场景 | 处理方式 |
|------|---------|
| 未抽签就进入环节 | 进入第一/四/五环节时检测 `draw.teamOrder` 是否非空；为空则弹出警告，阻止继续 |
| 图片分配有重复 | 手动模式下，提交前校验所有队图片编号唯一；若重复则提示并阻止锁定 |

### 13.8 强制覆盖出场顺序（应急暗门）

#### 背景

正常流程下，抽签锁定后 `draw.orderLocked = true`，所有修改入口均禁用。但现场可能出现例外情况（设备故障、选手临时缺席等），评委要求调换两队出场位置，若走完整解锁→重抽→重新确认流程，会耽误赛程、引发混乱。

为此，在设置面板提供一个**极高权限的强制覆盖入口**，绕过锁定直接改写 `draw.teamOrder`，并在流水中留存操作记录供事后审计。

#### 入口设计（故意设置操作门槛）

入口位于设置面板最底部「高级 / 危险操作」区，默认折叠，需**连续点击区域标题 3 次**才展开，防止误触：

```
┌──────────────────────────────────────────────────────────┐
│  ⚠️ 应急操作（点击标题 3 次展开）                           │
│  ──────────────────────────────（折叠，默认不可见）─────── │
│                                                          │
│  强制覆盖出场顺序                                         │
│  此操作绕过抽签锁定，直接修改出场顺序，请谨慎使用。          │
│                                                          │
│  操作人备注（必填）：[________________]                    │
│                                                          │
│  当前顺序：红队 · 绿队 · 蓝队 · 黄队 · 紫队               │
│                                                          │
│  新顺序（拖拽排序或下拉框选择）：                           │
│  1号位[红队▼]  2号位[蓝队▼]  3号位[绿队▼]  4号位[黄队▼]  5号位[紫队▼]│
│                                                          │
│  [取消]                       [强制覆盖（不可撤销）]       │
└──────────────────────────────────────────────────────────┘
```

**操作门槛**：
1. 需连续点击折叠区标题 3 次才展开（计数 5 秒内重置）
2. 「操作人备注」为必填项，为空时「强制覆盖」按钮禁用
3. 点击「强制覆盖」后弹出二次确认对话框：
   > 确认强制覆盖出场顺序？此操作将绕过抽签锁定，且不可撤销。请确保已获评委组授权。

#### 核心函数

```js
/**
 * 强制覆盖出场顺序（绕过锁定）
 * @param {Number[]} newOrder  新的 team.id 顺序数组
 * @param {String}  operator  操作人备注（必填，写入 log）
 */
function forceOverrideTeamOrder(newOrder, operator) {
  if (!operator || !operator.trim()) {
    throw new Error('操作人备注不能为空');
  }
  if (newOrder.length !== state.teams.length) {
    throw new Error('队伍数量不匹配');
  }
  // 校验：newOrder 必须是 team.id 的全排列
  const validIds = new Set(state.teams.map(t => t.id));
  const inputIds = new Set(newOrder);
  if (![...validIds].every(id => inputIds.has(id))) {
    throw new Error('出场顺序包含无效的队伍 ID');
  }

  const prev = [...state.draw.teamOrder];
  state.draw.teamOrder = newOrder;
  // 注意：不修改 orderLocked，锁定状态保持不变
  state.draw.log.push({
    type: 'force_override',
    prev,
    result: [...newOrder],
    operator: operator.trim(),
    ts: Date.now(),
  });
  save();
}
```

#### 对进行中环节的影响

`draw.teamOrder` 被覆盖后，**各环节的 `currentTeamIdx` 保持不变**，下一次推进时
自然读取新顺序中的对应位置。

> ⚠️ **注意**：设计稿此处写的 `advanceTeam()` **不存在**。真实的推进入口是 `index.html` 的
> `r1NextAnswerer()`（第一环节，按 4 人一组换队）与 `r4Next()` / `r4Prev()`（第四环节）；
> 两者都只改 `currentTeamIdx`，队伍实体经 `getTeamByIdx()` 现读 `draw.teamOrder` 取得。

| 场景 | 行为 |
|------|------|
| 强制覆盖发生在环节开始前 | 新顺序从头生效，无影响 |
| 强制覆盖发生在第一环节进行中 | 已出场队伍不受影响，**从下一支未出场队伍起**按新顺序进行 |
| 强制覆盖发生在第四环节进行中 | 同上，已找茬完的队不受影响 |
| 强制覆盖发生在第五环节进行中 | `r5.teamOrder` 是由 `draw.teamOrder` 在环节开始时复制的独立副本，**飞花令轮次顺序不受覆盖影响**（飞花令有独立的轮次逻辑） |

> **注意**：第五环节轮次顺序（`r5.teamOrder`）在环节开始时已从 `draw.teamOrder` 复制为独立副本，强制覆盖 `draw.teamOrder` 不会改变飞花令当前令题的轮次。如需同时调整飞花令轮次，需单独提供 `r5.teamOrder` 的覆盖入口（在第五环节操作面板内）。

## 十四、翻牌选题功能

### 14.1 设计理念

**问题**：纯随机出题虽然公平，但选手无参与感，且对结果容易产生质疑（"为什么偏偏给我出这道难题"）。

**解决方案**：翻牌选题——题目预先随机打乱并隐藏在若干背面相同的"牌"里，选手自主选择翻开哪张牌，系统按选手选择揭示题目。**本质是随机，体验是自主选择**，兼顾公平性与参与感，消除争议。

**适用环节**：

| 环节 | 翻牌对象 | 说明 |
|------|---------|------|
| 第一环节（个人必答） | 题目牌 | 每位选手从剩余牌池中自选 1 张，共 40 张牌 |
| 第四环节（识图找茬） | 图片牌 | 每队翻开 1 张，揭示本队对应的场景图 |

---

### 14.2 核心机制

**牌的本质**：每张牌是一个壳，背面一样，正面绑定一道题（或一张图/一个令题主题）。牌的排列在环节开始时一次性随机打乱，之后固定不变。选手选哪张、就翻哪张，内容由系统预先绑定。

```
环节开始时：
  questions（已按该环节随机打乱）
      ↓ 一对一绑定
  cards[0] cards[1] cards[2] ... cards[N]
  （全部 faceDown，背面相同，大屏网格展示）
      ↓ 选手/主持人点击某张
  cards[i].revealed = true
  → 翻牌动画 → 显示题目内容
  → TTS 朗读题目
  → 开始计时
```

---

### 14.3 各环节翻牌细节

#### 第一环节（个人必答）· 40 张牌

> `initCardDeck(1)` 中 `fisherYates(pool).slice(0, 40)` 取 40 张，
> 满足 5 队 × 4 人 × 2 题 = 40 次选牌。大屏按 10 列 × 4 行渲染（`display.html` 的 `CF_CFG[1]`）。
> （2026-07-16 前曾为 `slice(0, 20)`，已修复，见第十六章第 1 项。）

```
环节开始 → 生成 40 张牌（绑定 40 道题，位置随机打乱）
    ↓
大屏展示 10×4 牌面网格
所有牌正面朝下，编号 1~40 仅供定位
    ↓
轮到某选手时：
  控制台显示：「请{选手名}选牌（点击牌号或由选手口报牌号）」
  大屏高亮当前选手所在队颜色
    ↓
选手口头报出牌号（如"我选 17 号"）
  → 主持人在控制台点击对应牌
    ↓
翻牌动画（约 0.8 秒）→ 正面显示题目
  → TTS 朗读题目 + 选项
  → 开始 15 秒计时
    ↓
判分后，该牌标记为已翻（半透明灰显），选手不可再选
```

**牌面视觉状态**：
- 未翻：背面（统一图案，如品牌 Logo 或"?"）
- 当前选中（动画中）：翻转过渡态
- 已翻/已用：正面朝上半透明，隐约可见题号，不显示题目内容
- 当前展示：正面朝上，高亮边框

#### 第四环节（识图找茬）· 5 张图片牌

```
大屏显示 5 张背面相同的牌（对应 5 张场景图）
（图片已由赛前抽签分配给各队，但内容未揭示）
    ↓
轮到某队时：
  该队对应的牌自动高亮（因为抽签已决定哪张是哪队的）
  队代表翻牌（主持人点击）
    ↓
翻牌动画 → 场景图全屏展示
  → TTS：「请{队名}开始找茬，计时六十秒」
  → 60 秒计时开始
```

### 14.5 核心函数

```js
/**
 * 为指定环节生成牌组
 * round: 1 | 2 | 4    （第 2 环节也已接入，设计稿只写了 1|4）
 * 返回 false = 该环节未开启翻牌（cardFlip.rounds[round] 为假）
 */
function initCardDeck(round) {
  if (!state.cardFlip.rounds[round]) return false;
  let cards = [];
  if (round === 1) {
    const pool = state.questions.filter(q => q.round === 1);
    cards = fisherYates(pool).slice(0, 20).map((q, i) => ({   // ⚠️ 取 20 张，非 40
      id: `r1_${i}`, cardNum: i + 1, qId: q.id, revealed: false, used: false,
    }));
  } else if (round === 2) {
    const pool = state.questions.filter(q => q.round === 2);
    cards = fisherYates(pool).map((q, i) => ({                // 全部 4 张
      id: `r2_${i}`, cardNum: i + 1, qId: q.id, revealed: false, used: false,
    }));
  } else if (round === 4) {
    cards = state.draw.teamOrder.map((teamId, i) => ({        // 依赖抽签结果
      id: `r4_${i}`, cardNum: i + 1, teamId,
      imageKey: state.draw.r4ImageMap[teamId] || null,
      revealed: false, used: false,
    }));
  }
  state.cardFlip.cards   = cards;
  state.cardFlip.context = { round, teamId:null, memberIdx:null, pickCount:0, picked:[] };
  state.cardFlip.flipPulse = 0;
  state.cardFlip.lastFlippedCard = null;
  state.displayMode = 'cardflip';        // 自动切大屏到翻牌视图
  save();
  return true;
}

/**
 * 翻开某张牌 —— ⚠️ 这一步就已经翻开了，不是"翻开前的确认"
 * 返回 card 对象（不是 boolean）
 */
function selectCard(cardId) {
  const card = state.cardFlip.cards.find(c => c.id === cardId);
  if (!card || card.revealed || card.used) return false;
  card.revealed = true;                                     // 立即翻开
  state.cardFlip.lastFlippedCard = { ...card };
  state.cardFlip.flipPulse++;                               // 触发大屏动画
  state.cardFlip.context.picked.push(cardId);
  state.cardFlip.context.pickCount++;
  save();
  return card;
}

/**
 * 确认使用已翻开的牌 → 把题目绑定到对应环节的 currentQIdx
 */
function confirmFlip(cardId) {
  const card = state.cardFlip.cards.find(c => c.id === cardId);
  if (!card || !card.revealed) return false;                // 必须先 selectCard
  card.used = true;
  const round = state.cardFlip.context.round;
  if (round === 1 && card.qId != null) {
    state.r1.currentQIdx = state.questions.findIndex(q => q.id === card.qId);
  } else if (round === 2 && card.qId != null) {
    state.r2.currentQIdx = state.questions.findIndex(q => q.id === card.qId);
  } else if (round === 4 && card.imageKey != null) {
    state.r4.currentQIdx = state.questions.findIndex(q => q.imageKey === card.imageKey);
  }
  state.displayMode = 'question';        // 切回题目视图
  save();
  return true;
}

/** 未翻且未用的牌 */
function getAvailableCards() {
  return state.cardFlip.cards.filter(c => !c.revealed && !c.used);
}
```

#### ⚠️ 与设计稿的差异

| 设计稿 | 实际 |
|-------|------|
| 牌字段 `{position, faceValue, skipped, pickedBy}` | **`{cardNum, qId 或 teamId+imageKey, used}`**——`position`/`faceValue`/`skipped`/`pickedBy` **都不存在** |
| 牌 id `r1_card_00` | **`r1_0`**（无补零，无 `card`） |
| `selectCard()` 只做选中标记，`confirmFlip()` 才翻开 | **`selectCard()` 直接翻开并触发动画**；`confirmFlip()` 只是标记 `used` 并绑定题目 |
| `selectCard()` 返回 boolean，且 `picked` 被整体替换 | 返回 **card 对象**；`picked` 是 **push 追加**（可累积多张） |
| `confirmFlip()` 记录 `pickedBy` | ❌ 不记录。**谁翻的牌没有留痕** |
| 调 `getQuestionsForRound()` / `buildFaceValue()` / `fisherYatesShuffle()` / `activateFlippedQuestion()` | ❌ 这 4 个函数**都不存在**。实际用 `state.questions.filter()` + `fisherYates()`，题目绑定内联在 `confirmFlip()` 里 |
| 第一环节 40 张牌 | ✅ 取 40 张（`slice(0, 40)`，2026-07-16 修复；此前只取 20 张不够全场用） |
| 翻牌仅 1、4 环节 | **1、2、4 三个环节**（`cardFlip.rounds = {1:true, 2:true, 4:true}`） |
| `getAvailableCards()` 只过滤 `revealed` | 同时过滤 `revealed` 和 `used` |

### 14.8b 两个入口的分工：环节标签 vs [开始本环节]

⚠️ **这是最容易误解的一处**，2026-07-17 修正。

| 入口 | 函数 | 做什么 |
|------|------|--------|
| 顶部环节标签「① 个人必答」 | `switchPanel(n)` | 切面板 + 设 `currentRound` + 同步大屏视图；**场上无任何牌组时会顺手生成本环节牌组** |
| 面板内 `[开始本环节]` | `r1Init()` ~ `r5Init()` | 前置校验 + **重置本环节状态** + **强制重建牌组** |

修正前 `switchPanel()` 只设 `currentRound` 却不生成牌组，还把 `displayMode` 设成
`'question'`——于是点了环节标签后，顶栏徽章、大屏标题都已是「个人必答」，
**但大屏只显示"等待出题"**，主持人不知道还要再点一次 `[开始本环节]`。

现在的规则：

```js
if (usesFlip && deckForN)        → 回到翻牌，保留已翻/已用状态
else if (usesFlip && !anyDeck)   → initCardDeck(n) 自动生成（仅限场上无牌组时）
else                             → displayMode='question'，并在面板给出黄色提示
```

> ⚠️ **为什么「已有别的环节牌组」时不自动重建？**
>
> `state.cardFlip` **只存一副牌**（`cards` + `context.round`）。生成新牌组会覆盖旧的，
> 连同已翻/已用记录一起抹掉——**会导致重复出题**。
>
> 所以「随手点个标签」绝不能变成破坏性操作。此时面板顶部会显示黄色提示，
> 说明当前场上是哪个环节的牌组、以及点 `[开始本环节]` 会替换掉它。
> 该提示由 `renderDeckHint()` 生成，挂在 `render()` 上（不能挂 `switchPanel()`——
> `r2Init()` 是先 `switchPanel` 再 `initCardDeck`，挂前者会在牌组建好前就算出提示）。
>
> 单副牌模型的局限见 16.3 第 34 项。

### 14.9 环节启动的前置检查（`requireReady`）

✅ **2026-07-17 新增。** 进入任一环节前，`index.html` `requireReady(round, opts)` 统一校验，
不通过则 `alert` 说明**原因 + 解决位置**并自动跳回赛前准备面板。

| 检查 | 触发条件 | 提示 |
|------|---------|------|
| 出场顺序 | `draw.teamOrder` 为空（`opts.needDraw !== false` 时才查） | 「请先完成出场顺序抽签。位置：赛前准备面板 → [抽签]」 |
| 题库已导入 | `state.questions` 为空 | 「尚未导入题库…位置：赛前准备面板 → [导入题库 JSON]」 |
| 本环节有题 | `countQuestions(round) === 0` | 指出总题数与该 round 的题数，提示确认导入的是完整 `questions.json` |
| R4 图题分配 | `draw.r4ImageMap` 为空（仅 `r4Init()` 额外查） | 「位置：赛前准备面板 → [随机分配图题]」 |

各环节的 `needDraw`：R1 / R4 / R5 需要出场顺序；**R2（五队同答）与 R3（抢答）不需要**。

顶栏另有常驻告警 `#no-qbank-warn`：题库未导入时显示红色「⚠ 尚未导入题库」，点击跳回 P0，
导入后自动消失。因为 P0 的红字在切到其他面板后就看不见了。

> ⚠️ **注意：这修的是一个真实踩过的坑。**
>
> 此前题库未导入时点「个人必答」，`initCardDeck(1)` 会筛出空题池、造出
> **`displayMode='cardflip'` 但 `cards=[]`** 的矛盾状态，还**返回 `true`**。
> 大屏的判定是 `mode === 'cardflip' && state.cardFlip.cards?.length`，空数组为假，
> 于是**静默掉回 `updateQuestion()` 显示"等待出题"**——主持人在台上只看到"等待出题"，
> 完全无从判断是题库没导入。
>
> 现在 `initCardDeck()` 空牌组**直接返回 `false` 且不改动任何 state**，

### 15.2 计时核心设计

**原则**：用 `Date.now()` 时间戳而非递减变量驱动计时，两个窗口（控制台 + 大屏）各自用
`requestAnimationFrame` 独立渲染，不依赖跨窗口同步每一帧——只同步"启动/暂停/结束"事件，
避免 BroadcastChannel 延迟导致大屏倒计时飘移。

```
剩余时间 = durationMs − (Date.now() − startedAt) − elapsedMs（已暂停时长）
```

> ⚠️ **警告：`startTimer()` 的第一参是【毫秒】，外部调用必须自己乘 1000。**
>
> ```js
> startTimer(15, 3);                        // ✗ 15 毫秒，倒计时瞬间归零
> startTimer(state.r3.timerSec * 1000, 3);  // ✓ 15 秒
> ```
>
> 这是全文档最容易踩的坑：设计稿签名是 `startTimer(durationSec, ...)`，
> 实现是 `startTimer(durationMs, round = null)`——**单位变了，参数个数也变了**
> （无第三个 `autoExpire` 参数）。误传秒不会报错，只表现为"倒计时一闪而过"。
> 完整签名对照见 15.3。

**状态转移**：

```
idle
 ├─ [startTimer()] ──────────────────→ running
running
 ├─ [pauseTimer()] ──────────────────→ paused
 ├─ 剩余时间 ≤ 0 ────────────────────→ expired
 │     （由控制台 renderTimer() 检出：先 expireTimer() 置态，
 │       再调 onTimerExpired(round) 做各环节自动判定，见 15.4）
 └─ [resetTimer()] ──────────────────→ idle
paused
 ├─ [resumeTimer()] ─────────────────→ running
 └─ [resetTimer()] ──────────────────→ idle
expired
 └─ [resetTimer()] ──────────────────→ idle
```

> ⚠️ **注意**：设计稿此处写的回调名 `onExpire(round)` **不存在**，
> 真实函数是 `onTimerExpired(round)`，且它定义在 **`index.html`** 而非 `shared.js`
> （因为要调用 `doR1()` 等 UI 层函数）。大屏虽然也轮询渲染，但**不会**触发它——
> `expireTimer()` 只在控制台侧被调用，符合"大屏只读"的设计。

### 15.3 核心函数（shared.js）

> ⚠️ **最容易踩的坑：`startTimer()` 的第一个参数是毫秒，不是秒。**
> 设计稿写的是 `startTimer(durationSec, ...)`，实现是 `startTimer(durationMs, ...)`。
> 调 `startTimer(15)` 会得到 **15 毫秒**而不是 15 秒。
> 所有调用点都必须自己乘 1000：`startTimer(state.r3.timerSec * 1000, 3)`。

```js
/**
 * 启动倒计时
 * @param {number} durationMs  限时【毫秒】 ← 注意单位
 * @param {number} round       当前环节
 */
function startTimer(durationMs, round = null) {              // shared.js
  state.timer = {
    state: 'running',
    durationMs,                      // 直接存入，不做 ×1000
    startedAt: Date.now(),
    pausedAt: null,
    elapsedMs: 0,
    round,
  };
  save();
}

/** 暂停（累加已耗时长） */
function pauseTimer() {
  if (state.timer.state !== 'running') return;
  state.timer.elapsedMs += Date.now() - state.timer.startedAt;
  state.timer.pausedAt   = Date.now();
  state.timer.state      = 'paused';
  save();
  // 注：不把 startedAt 置 null（设计稿会置 null）。getRemainingMs() 靠 state 分支
  //     判断，不读残留的 startedAt，因此无副作用。
}

/** 继续 */
function resumeTimer() {
  if (state.timer.state !== 'paused') return;
  state.timer.startedAt = Date.now();
  state.timer.pausedAt  = null;
  state.timer.state     = 'running';
  save();
}

/** 重置 —— ⚠️ 只改 4 个字段，不重置 durationMs / round */
function resetTimer() {                                      // shared.js
  state.timer.state     = 'idle';
  state.timer.startedAt = null;
  state.timer.elapsedMs = 0;
  state.timer.pausedAt  = null;
  save();
}

/** 标记超时 —— ⚠️ 无守卫；只改状态，超时判定由控制台 renderTimer() 调 onTimerExpired() */
function expireTimer() {                                     // shared.js
  state.timer.state = 'expired';
  save();
}

/** 实时计算剩余毫秒（纯计算，不写 state） */
function getRemainingMs() {
  const t = state.timer;
  if (t.state === 'idle' || t.state === 'expired') return 0;
  let elapsed = t.elapsedMs;
  if (t.state === 'running' && t.startedAt) elapsed += Date.now() - t.startedAt;
  return Math.max(0, t.durationMs - elapsed);
}

/** 停止（内部辅助）—— ⚠️ 不调 save()，调用方须自行 save */
function stopTimer() {
  state.timer.state = 'idle';
}
```

#### ⚠️ 与设计稿的差异

| 设计稿 | 实际 |
|-------|------|
| `startTimer(durationSec, round, autoExpire = true)` | **`startTimer(durationMs, round = null)`**——单位是毫秒，无 `autoExpire` 参数（该字段已于 2026-07-16 删除） |
| `expireTimer()` 有 `state !== 'running'` 守卫 | 无守卫，任何状态都能置 `expired` |
| `expireTimer()` 调 `onTimerExpired(round)` | ✅ 2026-07-16 起：`renderTimer()` 归零时先记下 `timer.round`，调 `expireTimer()` 后再调 `onTimerExpired(round)`（`index.html`，仅控制台侧） |
| `resetTimer()` 整体重建 timer 对象 | 只改 4 个字段，`durationMs` / `round` 残留 |

#### 归零时发生的两件事

需要区分两件事（v3.0 时只有 ①，v3.1 补上了 ②）：

**① 状态翻转 —— ✅ 已实现（自动）**

控制台的 `requestAnimationFrame` 轮询 `renderTimer()`（由 `index.html` `timerLoop()`
驱动）在归零时会自动调 `expireTimer()`：

```js
// renderTimer() 内
if (ts === 'running' && rem === 0) {
  const round = state.timer.round;
  expireTimer();      // state.timer.state = 'expired'
  cdSpoken.clear();
  onTimerExpired(round);
}
```

所以 `timer.state` **会**自动变成 `'expired'`，大屏也会同步显示为 0。

**② 超时判分 —— ✅ 已实现（2026-07-16，见 15.4）**

`onTimerExpired(round)` 在控制台侧自动触发各环节超时判定。R3 的手动"超时"按钮
（`doR3Timeout()`）仍保留，供主持人裁量提前判超时。

> ⚠️ **注意**：注意 `expireTimer()` **只在控制台侧被调用**（`renderTimer()` 是控制台的函数）。
> 大屏虽然也轮询显示，但不会写状态——这符合"大屏只读"的设计。

### 15.4 超时回调（onTimerExpired）—— ✅ 已实现（2026-07-16）

`onTimerExpired(round)` 定义在 **`index.html`（控制台侧）**，由 `renderTimer()` 在倒计时
归零时调用（大屏只读，不触发判定）。各环节实际行为：

| 环节 | 超时自动行为 |
|------|-------------|
| R1 | 播报"时间到，作答超时"，播完后自动 `doR1(false)` 判 0 分（有 `currentQIdx != null` 守卫；主持人已提前判分则计时器早已停止，不会重复触发） |
| R2 | 只播报"时间到，请各队举板"，判分仍由主持人逐队录入 |
| R3 | 走 `r3AutoTimeout()`：`locked` 状态 → 按 `q.score_wrong`（兜底 -2）扣分记 `reason:'timeout'`，未用过补抢则播报后自动 `r3OpenSupplement()`，否则回 `idle`；`armed` 状态 → 播报"时间到，无人抢答"并回 `idle` |

R3 的手动"超时"按钮（`doR3Timeout()`）保留，供主持人提前裁定。

> ⚠️ **注意**：注意：若比赛现场不希望某环节自动判定（如 R1 想给选手宽限），主持人可在归零前

### 15.6 TTS 倒计时播报节点

✅ **功能已实现**，但没有独立的 `checkTTSCountdown()` 函数——逻辑内联在 `renderTimer()`
中，用 `cdSpoken` Set 去重（而非 `lastAnnouncedSec`）：

```js
// index.html 顶部
const cdSpoken = new Set();

// index.html —— renderTimer() 内
if (state.tts.readCountdown && state.tts.enabled) {
  const secs = Math.ceil(rem / 1000);
  const at   = state.tts.countdownAt || [10, 5, 3, 2, 1];
  if (at.includes(secs) && !cdSpoken.has(secs)) {
    cdSpoken.add(secs);

  }
}
```

> ⚠️ **与设计的差异**：

| 设计 | 实际 |
|------|------|
| 播报 `还剩${sec}秒。` | **只念数字本身**：`speak(String(secs))` → "10"、"5"、"3"… |
| 归零时播报"时间到。" | ❌ **不播报**。归零只调 `expireTimer()` + `cdSpoken.clear()` |
| `COUNTDOWN_AT` 常量 | 读 `state.tts.countdownAt`（默认 `[10,5,3,2,1]`），**设置面板可改** |
| `lastAnnouncedSec` 去重 | **`cdSpoken` Set** 去重 |

**`cdSpoken` 的清空机制**（2026-07-16 改进）：`renderTimer()` 每帧比对
`state.timer.startedAt` 与模块级的 `_lastTimerStart`，**只要计时器被重新启动就自动
`cdSpoken.clear()`**：

```js
if (state.timer.startedAt !== _lastTimerStart) {
  _lastTimerStart = state.timer.startedAt;

```

这一层覆盖**所有** `startTimer()` 调用点，包括 `shared.js` 内部的调用

够不着 `cdSpoken`，此前只能靠调用方手动跟一句 `cdSpoken.clear()`，容易遗漏。
各处遗留的手动 `cdSpoken.clear()` 现在是冗余但无害的。

> ⚠️ **`cdSpoken` 是模块级变量，不在 state 里**，因此不跨窗口同步、刷新即丢。这不影响
> 功能（大屏本就不播报），但意味着**控制台刷新后当前这一轮的已播报记录会重置**，

```js
// display.html —— SVG 结构（52×52 容器，整体 rotate(-90deg) 让起点在 12 点方向）
// <circle cx="26" cy="26" r="23" stroke="#1e3a5f" stroke-width="4"/>          ← 底环
// <circle id="ring-arc" cx="26" cy="26" r="23" stroke="#3b82f6"
//         stroke-dasharray="144.51" stroke-dashoffset="0"/>                    ← 进度弧
//
// CIRCUMFERENCE = 2 × π × 23 ≈ 144.51

function updateTimer() {
  const rem = getRemainingMs();
  const ts  = state.timer.state;
  const arc = document.getElementById('ring-arc');
  const txt = document.getElementById('ring-text');

  if (ts === 'idle')    { /* 灰环 + "--" */ return; }
  if (ts === 'expired') { /* 红环填满 + "0" */ return; }

  const dur   = state.timer.durationMs || 1;
  const pct   = rem / dur;
  const color = pct > .4 ? '#22c55e' : pct > .2 ? '#f59e0b' : '#ef4444';  // 绿→黄→红
  arc.style.stroke = color;
  arc.setAttribute('stroke-dashoffset', String(CIRCUMFERENCE * (1 - pct)));
  txt.textContent  = fmtMs(rem);
  txt.className    = 'ring-text' + (rem <= 5000 ? ' danger' : rem <= 10000 ? ' warn' : '');
}
```

两套阈值口径**不一致**，注意区分：
- **环颜色**按剩余**比例**：>40% 绿、>20% 黄、否则红
- **数字颜色**按剩余**绝对秒数**：≤5s 红（danger）、≤10s 黄（warn）

│      ┌──────┐                                            │
│      │  12  │  ████████████░░░░░░  12 / 15 秒            │
│      └──────┘                                            │
│                                                          │
│  [开始计时]  [暂停]  [继续]  [重置]                        │
└──────────────────────────────────────────────────────────┘
```

**按钮状态规则**：

| 计时器状态 | 开始 | 暂停 | 继续 | 重置 |
|-----------|------|------|------|------|
| idle | 可用 | 禁用 | 禁用 | 禁用 |
| running | 禁用 | 可用 | 禁用 | 可用 |
| paused | 禁用 | 禁用 | 可用 | 可用 |
| expired | 禁用 | 禁用 | 禁用 | 可用 |

**快捷键**：

| 快捷键 | 功能 |
|--------|------|
| `T` | 开始计时（idle 状态）/ 暂停（running 状态）/ 继续（paused 状态） |
| `Shift+T` | 重置计时器 |

---

### 15.9 各环节计时器触发时机

| 环节 | 何时启动计时器 | 何时自动重置 |
|------|-------------|------------|
| 第一环节 | 翻牌动画结束 / TTS 读题完毕（`onend` 回调）后自动启动 | 判分完成后自动重置，进入下一人 |
| 第二环节 | 主持人点击「开始」后手动触发（等全队就绪） | 超时或主持人点「亮板」后重置 |
| 第三环节 | 抢答队伍选定答题成员后自动启动 | 判分后自动重置，回到 idle |
| 第四环节 | 翻牌动画结束后自动启动 | 超时或主持人点「结束找茬」后重置 |
| 第五环节 | 轮到某队时自动启动 | 有效回答 / 出局 / 超时后自动重置，进入下一队 |

### 15.10 防错与边界

| 场景 | 设计 | 实际 |
|------|------|------|
| 计时运行中切换环节 | `switchRound()` 内调 `resetTimer()` | ✅ 等效实现：`initR1()`~`initR5()` 每个都调了 `resetTimer()`（`switchRound()` 本身不存在） |
| 计时运行中设备刷新 | `startedAt` 持久化，刷新后仍能正确计算 | ✅ 成立。但**超时不会自动 expired**，只是显示 0 |
| 主持人手动提前结束 | 调 `resetTimer()` 后再判分 | ⚠️ 实际用 `stopTimer()`（置 idle，不 save），由判分函数负责 save |
| 多次快速点击「开始计时」 | `startTimer()` 先 `resetTimer()` 再启动 | ⚠️ `startTimer()` **整体重建 `state.timer` 对象**，效果等同重置，不叠加。但它不调用 `resetTimer()` |
| 暂停期间 TTS 正在播报倒计时 | `checkTTSCountdown` 停止检测，恢复后重置 `lastAnnouncedSec` | ⚠️ 实际用 `index.html` 的 `cdSpoken` Set 去重，各处 `cdSpoken.clear()` 手动清空。无 `checkTTSCountdown` / `lastAnnouncedSec` |
