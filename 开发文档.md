## 一、项目概述

### 1.1 赛制背景

- **参赛规模**：5 支队伍，每队 4 人
- **五大环节**：个人必答 → 团队共答 → 擂台抢答 → 识图找茬 → 服务飞花令
- **计分结构**：基本盘（封顶 50 分）+ 浮动环节（无上限·可负）= 总分不封顶，最高者胜
- **预计用时**：约 105 分钟

### 1.2 实现现状

重写已完成。系统由两个页面 + 一个共享状态模块构成，围绕五大环节的赛制实现：

| 能力 | 状态 | 实现位置 |
|------|------|---------|
| 五环节状态机（`currentRound` / `roundPhase`） | ✅ | `shared.js` `initR1()`~`initR5()` |
| 七种题型判分 | ✅ | `shared.js` 各 `scoreRx()` |
| 抢答（含读题期违规扣分、补抢） | ✅ | `shared.js` `r3TryBuzz()` / `r3EarlyBuzz()` / `r3OpenSupplement()` |
| TTS 题目朗读 + 语音报分 | ✅ | `shared.js` TTS 模块（264–393 行） |
| 倒计时（跨窗口无漂移） | ✅ | `shared.js` 计时器模块（400–455 行） |
| 抽签（出场顺序 / 图题分配 + 锁定 + 强制覆盖暗门） | ✅ | `shared.js` 抽签模块（462–527 行） |
| 翻牌选题（第 1/2/4 环节） | ✅ | `shared.js` `initCardDeck()` / `selectCard()` / `confirmFlip()` |
| 分环节上限控制 | ⚠️ 部分 | 见第六章：各环节内联，非统一函数 |
| 并列检测 / 加赛 | ❌ | 见第十六章 |

> 历史背景（v2.0 时的起点）：重写前是一个通用抢答器，只有单一题型（文本/图片/视频）、
> 抢答 + 手动判分，无环节概念。这段历史仅供理解 storage key 为何升到 `v3`，
> 现已无代码残留。

### 1.3 技术约束

- **纯前端**：不依赖任何服务器，无构建步骤，无 npm 依赖。
- **跨窗口同步**：控制台（`index.html`）与大屏（`display.html`）通过 BroadcastChannel
  实时广播 + localStorage 持久化双通道同步。
- **无框架**：Vanilla JS，CSS 自定义属性，不引入第三方库。
- **兼容性**：现代 Chromium 内核浏览器（Chrome 90+）。TTS 依赖 `speechSynthesis`。
- **页面角色标记**：`index.html` 在加载 `shared.js` 前设置 `window.IS_CONTROL = true`
  （见 `index.html:155`）。TTS 与计时器启动**仅在控制台侧执行**，大屏不发声，
  避免两个窗口同时朗读。`isTTSAvailable()` 以此为前置判断。

#### 运行方式

题库通过设置面板的文件选择框手动导入（`index.html` `loadQFile()`），代码中**没有**
`fetch('./questions.json')` 之类的自动加载，因此双击 HTML 用 `file://` 打开不会触发
CORS 问题，可直接运行。

开发调试时若希望免去每次手动导入，可跑一个静态服务器再自行扩展加载逻辑：

```bash
cd C:/Users/admin/Desktop/quiz_buzzer
python -m http.server 8080     # 然后访问 http://localhost:8080/index.html
```

状态持久化在 localStorage 的 `rz_contest_v3` 键下，刷新不丢；改动状态结构后如需清空，
在控制台执行 `localStorage.removeItem('rz_contest_v3')` 再刷新。

### 2.2 题库构建链路

`questions.json` **不是手写的**，由 `parse_qbank.py` 从 `raw_qbank.txt` 解析生成：

```

        │ （人工提取文本）
        ▼
   raw_qbank.txt
        │ python parse_qbank.py
        ▼
```

重跑会覆盖手改内容。脚本内的路径是硬编码绝对路径（`C:/Users/admin/Desktop/quiz_buzzer/`），

## 三、数据模型

### 3.1 队伍与成员

```js
// 队伍对象
{
  id: Number,          // 1~5，固定 ID
  name: String,        // "一队"~"五队"，可自定义
  color: String,       // 颜色（CSS 色值），展示用
  members: String[],   // 长度 4，成员姓名
  scores: {            // 各环节得分，key 为 ROUND_KEY

    r2: Number,        // 第二环节（团队共答）队伍合计
    r3: Number,        // 第三环节（擂台抢答）可正可负
    r4: Number,        // 第四环节（识图找茬）
    r5: Number,        // 第五环节（飞花令）
  },
  memberScores: {      // 个人必答环节 · 成员得分（key 为成员 index 0~3）
    0: Number,
    1: Number,
    2: Number,
    3: Number,
  }
}
```

### 3.2 题库结构

题库是一个扁平数组，**不按环节分组**，靠每题的 `round` 字段区分环节：

```js
{ "questions": [ /* Question[] */ ] }
```

**没有统一的题目对象**——不同题型的字段集差异很大，只有 4 个字段是全体共有的：

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | String | 唯一 ID，格式 `r{环节}_{类型缩写}_{序号}`，如 `r1_001` / `r3_tf_001` |
| `round` | Number | 所属环节 1–5 |
| `type` | String | 题型枚举（见 3.3） |
| `stem` | String | **题干文本**（注意：不是 `content`） |

其余字段按题型出现，各题型的完整字段见 **第七章 · 题库 JSON 格式规范**。

⚠️ **易踩的坑**：

- 题干字段是 **`stem`**，不是 `content`。代码全线读 `q.stem`（如 `buildQuestionSegments()`
  `shared.js:384`）。
- 分值字段是 **`score_correct` / `score_wrong`**（下划线命名），不是 `scorePerPoint`。
- `blanks` 是**对象数组** `[{hint, answer[]}]`，不是空数（Number）。
- **不存在** `title` / `content` / `media` / `spotCount` / `usedBy` / `flyingFlowerTheme`
  这些字段，代码里也没有任何地方读它们。
- `spot` / `theme` 题**没有** `score_correct`，其分值由代码规则决定（每找对 1 处 +1；
  飞花令有效 +1、擂主 +3），不从题库读。

### 3.3 题型枚举

代码与题库中实际存在 **7 种**题型（`index.html:930` `typeMap` 是权威列表）：

| 枚举值 | 中文名 | 使用环节 | 题量 | 有 `options` | 答案字段 |
|--------|-------|---------|------|-------------|---------|
| `single` | 单选 | 1、3 | 40 + 16 | ✅ | `answer: "B"` |
| `multi` | 多选 | 2、3 | 2 + 67 | ✅ | `answer: ["A","B","C"]` |
| `judge` | **判断** | 3 | 65 | ❌ | `answer: "√"` 或 `"×"` |
| `fill` | 填空（单空） | 3 | 68 | ❌ | `answer: "45"` |
| `fill_multi` | 多空填空 | 2 | 2 | ❌ | `blanks: [{hint, answer[]}]` |
| `spot` | 识图找茬 | 4 | 5 | ❌ | `spots: [{key, label}]` |
| `theme` | 服务飞花令 | 5 | 3 | ❌ | `answerPool: String[]` |

合计 268 题。

⚠️ **与 v2.0 设计稿的命名差异**（设计稿的名字在代码中一律不存在）：

| v2.0 设计稿 | 实际实现 |
|------------|---------|
| `fill_single` | `fill` |
| `spot_image` | `spot` |
| `flying_flower` | `theme` |
| （未设计） | **`judge`** ← 判断题，65 道，占第三环节近三成 |

`judge` 是设计稿完全遗漏、但代码与题库都已实现的题型。它没有 `options`，
`answer` 直接存 `√` / `×` 字符；控制台在 `renderQBox()`（`index.html:935`）中把它
与 `single`/`multi` 一同当作可交互选择题渲染，选项由 UI 侧提供而非题库提供。

#### 状态恢复的合并策略

`load()` 用 `deepMerge(defaultState(), loaded)` 恢复状态（`shared.js:223`），而非直接
覆盖。这意味着**新增 state 字段时老存档能自动补上默认值**，无需清 localStorage。但注意
`deepMerge` 只递归普通对象，**数组是整体替换**——所以往数组类字段加默认元素不会生效。

### 3.5 判分事件（ScoreEvent）

全部判分流水追加进 `state.history`，由 `logEvent(event, skipAnnounce)` 统一写入
（`shared.js:1141`）。**事件不是统一结构**，各环节按需附加字段。

**公共字段**（各环节都有）：

```js
{
  round: Number,      // 环节 1~5
  teamId: Number,
  teamName: String,
  correct: Boolean,   // 是否判为正确（注意：v2.0 未列出此字段，实际必有）
  delta: Number,      // 实际得分变化量（已经过上限裁剪，可正/负/零）
  reason: String,     // 见下表
  ts: Number,         // 时间戳
}
```

**按环节附加的字段**：

| 附加字段 | 类型 | 出现在 | 说明 |
|---------|------|-------|------|
| `memberIdx` / `memberName` | Number / String | R1、R3 | 个人得分归属；队伍整体得分（R2/R4/R5）**不带这两个字段** |
| `qId` | String | R1、R2、R3 | 题目 id |
| `fromBuzz` | Boolean | R3 | 恒为 `true`，标记来自抢答 |
| `foundCount` | Number | R4 | 实际找出的点数（裁剪前） |
| `answer` | String | R5 | 有效答案内容 |
| `eliminated` | Boolean | R5 | 出局事件标记 |
| `manual` | Boolean | `adjust` | 手动调分标记 |

⚠️ **不存在 `note` 字段**（v2.0 设计了但未实现，代码中 0 处引用）。备注信息目前无处存放。

**`reason` 取值**：

| 取值 | 触发处 | delta |
|------|-------|-------|
| `correct` | 各环节答对 | 依题型 |
| `wrong` | 各环节答错 / R5 出局 | R3 为 -2；R1/R2 为 0；R5 为 0 |
| `violation` | R3 读题期违规抢答 `r3EarlyBuzz()` | -2 |
| `timeout` | R3 超时，由 `index.html:1292` `doR3Timeout()` 直接构造 | -2 |
| `spot` | R4 结算 `scoreR4()` | 找对点数（≤10） |
| `flower_valid` | R5 有效答案 | +1 |
| `flower_winner` | R5 擂主 | +3 |
| `adjust` | 手动调分 `adjustScore()` | 任意 |

⚠️ `timeout` 是唯一**不由 `shared.js` 生成**的 reason——它在控制台 `doR3Timeout()` 里
手工拼装事件对象后调 `logEvent()`。业务规则外泄到了 UI 层，是已知的架构瑕疵（见第十六章）。

## 四、计分规则实现

### 4.1 第一环节 · 个人必答（单选）

**赛制规则**：
- 每人 2 题，单题 15 秒，口头作答
- 答对：+2.5 分（同时计入队伍分与个人分）
- 答错/超时：0 分，不扣分
- 违规（队内外提示）：本题 0 分 **且** 队伍扣 2 分 ← ⚠️ **未实现**
- 单队上限 20 分（4 人 × 2 题 × 2.5 分 = 20，即全对即封顶）

**实际实现**：

```js
function scoreR1(correct) {
  const team      = getTeamByIdx(state.r1.currentTeamIdx);  // 从 state 取，不传参
  const memberIdx = state.r1.currentMemberIdx;
  const q         = state.questions[state.r1.currentQIdx];
  const baseScore = q?.score_correct ?? 2.5;   // 分值来自题库，兜底 2.5
  const delta     = correct ? baseScore : 0;

  // 上限裁剪（内联，非统一函数）
  const current = team.scores.r1 || 0;
  const actual  = Math.max(0, Math.min(delta, ROUND_CAPS.r1 - current));

  team.scores.r1 += actual;
  team.memberScores[memberIdx] += actual;   // 个人分同步累加
  logEvent({ round:1, teamId:team.id, memberIdx, delta:actual,
             reason: correct ? 'correct' : 'wrong', qId:q?.id, ... });
}
```

> 实现位置：`shared.js:659`

**与设计稿的差异**：

| 设计稿 | 实际 |
|-------|------|
| `scoreR1(teamId, memberIdx, result)` 三参数 | `scoreR1(correct)` **单个布尔值**，队伍/选手/题目全部从 `state.r1` 读取 |
| `result` 三态含 `'violation'` | 只有 `true`/`false` 两态 |
| 违规 -2 分 | ❌ **未实现**——`scoreR1` 没有违规分支，第一环节违规目前只能用手动调分 `adjustScore(teamId,'r1',-2)` 补救 |
| 分值硬编码 2.5 | 读 `q.score_correct`（题库中 R1 全部为 2.5），兜底 2.5 |
| 走 `applyTeamScore()` 统一裁剪 | 函数内联 `Math.min`（见第六章） |

**上限行为**：`actual = Math.max(0, Math.min(delta, 20 - current))`，即**超出部分直接丢弃**，
队伍分与个人分加的是裁剪后的 `actual`。这意味着封顶后个人分也不再增长——若需保留个人
真实答对数用于评"最佳个人"，当前数据不支持（见第十六章）。

### 4.2 第二环节 · 团队共答（多选/多空）

**规则**：
- 5 队同时用答题板作答，限时 40 秒
- 单题 5 分
- **多选题**：正确选项集合完全一致得满分（按正确项均分）；任一错选→本题 0 分
- **多项填空**：按空均分 5 分，每空独立计分，错/漏不扣分
- 单队上限 20 分

**多选判分**：`scoreR2Multi(teamId, selected, correctOptions, totalScore = 5)`

```js
// 含任一错选 → 0 分；否则按"已选数 / 正确数"比例给分（漏选得部分分）
let hasWrong = [...selSet].some(s => !corrSet.has(s));
let delta = (!hasWrong && selSet.size > 0)
          ? (selSet.size / corrSet.size) * totalScore
          : 0;
const actual = Math.max(0, Math.round(Math.min(delta, ROUND_CAPS.r2 - current) * 100) / 100);
```

> 实现位置：`shared.js:715`。返回 ScoreEvent 对象（不是分数），并已写入 history。

注意 `event.correct` 与 `event.reason` 判定口径不同：
`correct` 要求**全对且无错选**（`!hasWrong && selSet.size === corrSet.size`），
而 `reason` 只要**无错选且有选**就记 `'correct'`（漏选也算）。即漏选场景下
`reason: 'correct'` 但 `correct: false`，语音会播报"答对"。这是已知的口径不一致。

**多空判分**：`scoreR2FillMulti(teamId, blanks, correctBlanks, totalScore = 5)`

```js
const perBlank = totalScore / correctBlanks.length;
let correct = 0;
correctBlanks.forEach((ans, i) => {
  const given    = (blanks[i] || '').trim();
  const expected = Array.isArray(ans) ? ans : [ans];   // 每空允许多个同义词
  if (expected.some(e => e.trim() === given)) correct++;
});
const delta = Math.round(correct * perBlank * 100) / 100;
```

> 实现位置：`shared.js:753`

⚠️ **答案比对只做 `trim()`，没有归一化**。设计稿提到的 `normalize()`（全半角、标点差异
处理）**不存在**，代码是严格字符串相等比较。因此"不出卖场"与"不出卖场。"会判为不同答案。
实际操作中由计分员肉眼比对后手工输入，风险可控，但若要做自动判分需补 `normalize()`
（见第十六章）。

### 4.3 第三环节 · 擂台抢答（混合题型）

**规则**：
- 系统 TTS 读题完毕后自动喊"开始抢答"，大屏同步显示"开始抢答"字样
- **大屏显示"开始抢答"之前**按下抢答器 = 违规，系统立即扣 2 分并语音播报
- **大屏显示"开始抢答"之后**第一个按下抢答器的队伍 = 抢答成功
- 抢到后全队可沟通，指派 1 人作答，限时 15 秒
- 答对：+2 分；答错/超时：-2 分；补抢：若首抢队答错/超时，可开放一次补抢（仅一次）
- 无上下限（可为负）

**抢答状态机**（`r3.buzzState`）：

实际只有 **4 个可达状态**：`idle` → `reading` → `armed` → `locked`。

```
idle
 └─ [主持人点"出题" r3StartQuestion()] ──────→ reading
      （TTS 朗读题干 + 选项，大屏不显示"开始抢答"）
      │
      │  [任意队按抢答器 → r3EarlyBuzz()] ──→ 违规 -2 分，打断 TTS 播报违规，
      │                                        然后 resumeR3Reading() 从头重读 → 仍是 reading
      │
 reading → [TTS 队列播完（末段为"开始抢答"）] → armed
      （同一 onend 回调里启动 15 秒计时器）
      │
      │  [任意队按抢答器 → r3TryBuzz()] ────→ locked
      │
 armed → locked
      （stopTimer()，TTS 播报"XX队抢答成功，请答题"）
      │  ⚠️ 抢到后计时器被停止，答题 15 秒限时未自动重启
      │
 locked → [r3Score(true)]  ─────────────────→ idle（+2，TTS 报分，自动亮答案+分数）
        → [r3Score(false)] ─────────────────→ idle（-2，TTS 报分）
        → [doR3Timeout()]  ─────────────────→ idle（-2，reason:'timeout'）
        → [r3OpenSupplement()] ─────────────→ armed（补抢，仅一次，重启计时器）
```

⚠️ **`'supplement'` 是死枚举值**：`defaultState()` 的注释里列了它
（`shared.js:74` `idle|reading|armed|locked|supplement`），但**代码中从未对 `buzzState`
赋过这个值**。补抢由 `r3OpenSupplement()` 实现，它把状态设回 **`'armed'`** 并置
`supplementUsed = true`。

因此 `index.html` 中这两处针对 `'supplement'` 的分支是**永不可达的死代码**：
- `index.html:793` —— `supplement: {text:'开放补抢！', ...}` 状态标签映射
- `index.html:804` —— `if (bs === 'locked' || bs === 'supplement')`

补抢是否可用，实际靠 `supplementUsed` 布尔值判断（`index.html:822` 控制按钮禁用）。
清理建议见第十六章。

**状态转换的守卫条件**：

| 函数 | 前置要求 | 不满足时 |
|------|---------|---------|
| `r3TryBuzz(teamId)` | `buzzState === 'armed'` | 若为 `'reading'` 则转 `r3EarlyBuzz()`；其余返回 `false` |
| `r3Score(correct)` | `selectedTeam` 且 **`selectedMember != null`** | 返回 `null`，不判分 |
| `r3OpenSupplement()` | `buzzState === 'locked'` 且 `!supplementUsed` | 返回 `false` |

⚠️ `r3Score()` **要求先指定答题人**（`r3SelectMember(memberIdx)`），否则静默返回 `null`
什么也不做。抢答成功后 `selectedMember` 被置 `null`，必须由主持人在控制台点选队员。

**大屏三阶段显示**：

| buzzState | 大屏中央显示 | 颜色 |
|-----------|------------|------|
| `reading` | 题目内容（正常展示，无特殊提示） | 默认 |
| `armed` | 题目内容 + 顶部横幅**「开始抢答」** | 金色横幅，脉冲动画 |
| `locked` | **「XX队 抢答成功」**大字覆盖（约 1.5 秒）→ 退回题目页 | 队伍颜色 |

**违规检测实现**：

**没有 `onBuzzerKeyPress()` 分发器**。控制台的键盘监听直接调 `r3TryBuzz()`，由它内部
按状态分发（`shared.js:837`）：

```js
// index.html:1049 —— 抢答器按键唯一入口

  if (curPanel !== 3) return;                      // 仅第三环节面板生效

  if (state.r3.buzzState !== 'armed')   return false;   // idle/locked 静默忽略
  ...

```

违规扣分（`shared.js:862`）：

```js
function r3EarlyBuzz(teamId, onDone) {
  if (window.IS_CONTROL) stopSpeak();            // 打断 TTS
  team.scores.r3 = (team.scores.r3 || 0) - 2;    // 硬编码 -2，无上限裁剪
  logEvent({ ..., delta:-2, reason:'violation' }, true);  // ← skipAnnounce=true

```

注意 `logEvent(event, true)` 的第二参 `skipAnnounce` 置 `true`，避免 `announceScore()`
与这里的 `speakQueue` 同时发声造成双重播报。

**判分函数**：`r3Score(correct)`（`shared.js:912`）

```js
const delta = correct ? 2 : -2;        // ⚠️ 硬编码，不读 q.score_correct / q.score_wrong
team.scores.r3 += delta;               // 无上限裁剪（R3 本就无上限）
team.memberScores[memberIdx] += delta; // 个人分同步（可为负）
if (correct) { showAnswerOnDisplay = true; showScoresOnDisplay = true; }  // 自动亮答案
stopTimer();
```

⚠️ **分值硬编码**：`r3Score()` 用字面量 `2` / `-2`，**不读题库的 `score_correct` /
`score_wrong`**。当前题库 R3 全部题目恰好是 `score_correct: 2, score_wrong: -2`，
所以结果一致——**这是巧合，不是设计**。一旦题库里出现不同分值的 R3 题目，判分会静默出错。
对比 `scoreR1()` 是正确读题库的。修复建议见第十六章。

**与设计稿的差异**：设计稿的 `scoreR3(teamId, memberIdx, correct, isViolation)` 不存在；
违规与判分是两个独立函数（`r3EarlyBuzz()` / `r3Score()`），且都从 `state.r3` 读上下文
而非传参。

### 4.4 第四环节 · 识图找茬

**规则**：
- 每队 1 张图（共 5 图，提前抽签分配）
- 每图预设 10 个找茬点，60 秒内口述
- 找对 1 处 +1 分，最高 10 分，不扣分
- 评委可临时认定预设之外的合理违规点（仍封顶 10 分）
- 超时未表述完的不计分；重复描述不重复计分

**评委端操作**：
- 10 个点位复选框（勾选=有效）
- 额外可输入自定义找茬点（评委认定后添加）
- 实时汇总分数（min(已勾选总数, 10)）

**判分函数**：`scoreR4(teamIdx)`（`shared.js:1005`）

注意参数是 **`teamIdx`（出场顺序下标）**，不是 `teamId`；勾选结果与额外点从
`state.r4` 读取，不传参：

```js
function scoreR4(teamIdx) {
  const team  = getTeamByIdx(teamIdx);            // 经 draw.teamOrder 转换
  const found = Object.values(state.r4.spotJudge).filter(Boolean).length
                + state.r4.extraSpots.length;
  const actual = Math.max(0, Math.min(Math.min(found, 10), 10 - (team.scores.r4 || 0)));
  team.scores.r4 += actual;
  logEvent({ round:4, ..., delta:actual, reason:'spot', foundCount:found });
}
```

`foundCount` 记录裁剪**前**的真实找出点数，`delta` 是裁剪后实际给分，便于赛后复盘。

**相关函数**：
- `r4JudgeSpot(spotKey, found)`（`shared.js:990`）—— 勾选单个点位。`spotKey` 是题库
  `spots[].key` 的字符串（如 `"a1"`），不是数字下标。
- `r4AddExtraSpot(desc)`（`shared.js:996`）—— 追加评委认定的额外点，存
  `{desc, ts}` 对象。

⚠️ **勾选数不设上限**：设计稿说"勾选框满 10 后禁止新增勾选"，代码**未实现**该限制，
`r4JudgeSpot()` 可无限勾选、`r4AddExtraSpot()` 可无限追加。只在 `scoreR4()` 结算时
clamp 到 10。多勾不会多得分，但 UI 上不会阻止。

### 4.5 第五环节 · 服务飞花令

**规则**：
- 共 3 个令题，每令题各队按顺序轮流接龙
- 每次 8~10 秒内说出 1 条有效内容（全队可商议）
- 有效（不重复、不违规）：+1 分
- 出局（重复/说错/超时）：退出本令题后续轮次，当次不扣分
- 本令题最后存活 1 队 = 擂主，额外 +3 分
- 若最终并列，以新令题加赛决胜

**状态机**（单令题）：
```
r5StartTheme(idx) → activeTeams = 全部 5 队，currentTurnIdx = 0
   │
   ├─ r5ValidAnswer(teamId, answer) → +1 分，记入 usedAnswers → _r5NextTurn()
   │
   └─ r5Eliminate(teamId) → 从 activeTeams 移除
           │
           ├─ 剩余 > 1 队 → _r5NextTurn()
           └─ 剩余 == 1 队 → 自动 r5SetWinner(最后一队) → +3 分
```

**判分函数**：

```js
// shared.js:1059 —— 有效答案，注意有第二参 answer
function r5ValidAnswer(teamId, answer) {
  state.r5.usedAnswers.push({ teamId, answer, ts: Date.now() });
  team.scores.r5 += 1;
  logEvent({ round:5, ..., delta:1, reason:'flower_valid', answer });
  _r5NextTurn();            // 自动推进到下一队
}

// shared.js:1078 —— 出局
function r5Eliminate(teamId) {
  state.r5.activeTeams = state.r5.activeTeams.filter(id => id !== teamId);
  logEvent({ ..., delta:0, reason:'wrong', eliminated:true }, true);  // skipAnnounce
  if (window.IS_CONTROL) speak(`${team.name}出局`);
  if (state.r5.activeTeams.length === 1) r5SetWinner(state.r5.activeTeams[0]);  // 自动决出擂主
  else _r5NextTurn();
}

// shared.js:1100 —— 擂主 +3
function r5SetWinner(teamId) {
  team.scores.r5 += 3;
  state.r5.themeWinners.push({ themeIdx: state.r5.currentThemeIdx, teamId });
  state.showScoresOnDisplay = true;      // 自动亮分
}
```

**轮次推进**（`_r5NextTurn()`，`shared.js:1117`）：从 `currentTurnIdx + 1` 起在
`r5.teamOrder` 上环形查找下一支仍在 `activeTeams` 中的队伍，带 `guard` 计数防死循环；
落位后**自动启动 10 秒计时器**。

⚠️ **答案去重未实现**：`usedAnswers` 只做记录，`r5ValidAnswer()` **不校验重复**——
传进来的答案即使与已有答案完全相同也照样 +1 分。防重复完全依赖评委肉眼比对
`answerPool` 与已用列表后决定调 `r5ValidAnswer()` 还是 `r5Eliminate()`。

⚠️ **`r5.timerSec` 被 `initR5()` 强制重置为 10**（`shared.js:1036`），即使用户在设置里
改过也会被覆盖。赛制说 8~10 秒，若要用 8 秒需在 `initR5()` 之后再改。

### 4.6 总分计算

```js
function teamTotal(team) {
  return Object.values(team.scores).reduce((a, b) => a + (b || 0), 0);
}
```

总分 = r1 + r2 + r3 + r4 + r5（r3 可为负），无封顶，最高者胜。

## 六、上限控制逻辑

```js
const ROUND_CAPS = {
  r1: 20,   // 个人必答上限 20 分
  r2: 20,   // 团队共答上限 20 分
  r3: null, // 抢答无上限（可负）
  r4: 10,   // 识图找茬上限 10 分
  r5: null, // 飞花令无上限
};

function applyTeamScore(teamId, roundKey, delta) {
  const team = getTeam(teamId);
  const cap = ROUND_CAPS[roundKey];
  const current = team.scores[roundKey] || 0;
  const next = current + delta;
  team.scores[roundKey] = cap != null ? Math.min(next, cap) : next;
  save();
}
```

## 十、重写要点与注意事项

### 10.1 与现有代码的关系

| 现有功能 | 重写处理 |
|---------|---------|
| localStorage 持久化 + BroadcastChannel 广播 | 保留，storage key 升级为 `v3` |
| 抢答器键位映射 | 保留，仅用于第三环节 |
| 队伍颜色配置 | 保留 |
| Logo / 品牌名 | 保留 |
| 题库 JSON 导入导出 | 扩展支持新字段 |
| 单一判分按钮 | 拆分为各环节独立操作区 |
| 题目列表（左侧） | 改为环节内题目进度，不再全局展示 |

### 10.2 关键实现细节

1. **第一环节出场顺序**：由抽签决定，在赛前设置中配置 `r1.teamOrder`，系统按此顺序驱动进度。

2. **第二环节同时亮板**：系统不自动判分，由计分员逐队对照举板内容，在控制台点击选项后手动提交。

3. **第三环节早抢检测**：`armed` 状态下按抢答器才合法；主持人尚未按"开始抢答"时，队伍按下直接触发 `r3EarlyBuzz`（-2 分，不锁定）。

4. **第四环节评委扩展点**：勾选框满 10 后禁止新增勾选，额外找茬点也累入总数（总量仍 clamp 到 10）。

5. **第五环节答案归一化**：`normalize(str)` 去除全半角、标点差异后再比较，同义词库由 `answerPool` 旁标注，人工最终确认。

6. **并列检测**：在第五环节结束后，`hasTie()` 检查前三名是否有影响排名的并列，自动触发加赛令题流程。

7. **分数清零确认**：清空操作弹出二次确认对话框，并在 history 中留存清零事件记录。

8. **抽签结果锁定**：主持人点击"确认抽签"后 `draw.isCompleted` 置 `true`，所有抽签按钮禁用；清空所有数据时同步解锁，允许重新抽签。

### 10.3 展示页自适应

- 大屏展示面向投影（1920×1080 或 1280×800），使用 `clamp()` 动态字体
- 题目内容超长时自动缩小字号
- 飞花令令题主题字号最大，醒目展示

## 十二、题目朗读系统（TTS）

### 12.1 设计背景与目标

比赛中主持人仅负责串场与流程推进，**题目宣读、选项朗读、环节过渡语、倒计时提示、抢答判定播报**等全部由系统通过浏览器内置 TTS（Web Speech API / SpeechSynthesis）自动完成，无需外接设备，无需后端。

**目标**：
- 解放主持人，减少人为宣读失误
- 语速、音色可配置，适配不同场地音响
- 所有朗读内容与系统操作事件绑定，触发准确、时机合理
- 控制台保留手动打断 / 重读 / 跳过按钮，应对突发情况

---

### 12.2 技术方案

**使用 Web Speech API（SpeechSynthesis）**，无需任何第三方库或后端接口。

```js
// 核心对象
const synth = window.speechSynthesis;

// 创建话语
function speak(text, options = {}) {
  synth.cancel();  // 打断当前正在播放的内容
  const utt = new SpeechSynthesisUtterance(text);
  utt.lang   = options.lang  ?? state.tts.lang;    // 默认 'zh-CN'
  utt.rate   = options.rate  ?? state.tts.rate;    // 默认 1.0（0.5~2.0）
  utt.pitch  = options.pitch ?? state.tts.pitch;   // 默认 1.0
  utt.volume = options.volume ?? state.tts.volume; // 默认 1.0
  utt.voice  = getTTSVoice();
  utt.onend  = options.onend ?? null;
  synth.speak(utt);
  return utt;
}

// 队列式朗读（多段依次播放）
function speakQueue(segments, onAllDone) {
  if (!segments.length) { onAllDone?.(); return; }
  const [first, ...rest] = segments;
  speak(first, { onend: () => speakQueue(rest, onAllDone) });
}
```

**兼容性**：Chrome 33+（含 Chromium 内核）原生支持，无需 polyfill。中文语音需操作系统安装有中文 TTS 引擎（Windows 10/11 默认已有「微软云希」/「微软晓晓」等）。

### 12.3 TTS 状态配置（state.tts）

```js
tts: {

  lang: 'zh-CN',          // 语言代码
  voiceName: String,      // 选定语音名称（空串=系统默认）
  rate: Number,           // 语速 0.7~1.5，默认 1.0
  pitch: Number,          // 音调 0.8~1.2，默认 1.0
  volume: Number,         // 音量 0~1，默认 1.0
  autoRead: Boolean,      // 切题时自动朗读题目（默认 true）
  readOptions: Boolean,   // 是否朗读选择题选项（默认 true）
```

#### 12.5.2 分值转中文朗读

TTS 直接朗读"2.5"会读成"二点五"，"10"会读成"十"，需要统一转换为自然中文，避免机械感：

```js
/**
 * 将分值数字转换为适合 TTS 朗读的中文字符串
 * 1 → "一"，2 → "两"，2.5 → "两点五"，10 → "十"
 */
function scoreToSpeech(n) {
  if (n === 0) return '零';
  const abs = Math.abs(n);
  // 整数部分
  const intPart = Math.floor(abs);
  const decPart = Math.round((abs - intPart) * 10);  // 最多一位小数
  const intMap = { 1:'一', 2:'两', 3:'三', 4:'四', 5:'五',
                   6:'六', 7:'七', 8:'八', 9:'九', 10:'十',
                   20:'二十' };
  const intStr = intMap[intPart] ?? String(intPart);
  if (decPart === 0) return intStr;
  const decMap = { 1:'一', 2:'二', 3:'三', 4:'四', 5:'五' };
  return `${intStr}点${decMap[decPart] ?? decPart}`;
}
// 示例：scoreToSpeech(2.5) → "两点五"
//        scoreToSpeech(10)  → "十"
//        scoreToSpeech(3)   → "三"
```

#### 12.5.3 报分函数

```js
/**
 * 判分完成后调用，自动生成并播报得分语音
 * @param {ScoreEvent} event  logEvent 产生的判分事件
 */
function announceScore(event) {
  if (!state.tts.enabled) return;

  const name = event.teamName;
  const delta = event.delta;
  let text = '';

  switch (event.reason) {
    case 'correct':
      text = `${name}，答对，得${scoreToSpeech(delta)}分。`;
      break;
    case 'wrong':
      text = delta < 0
        ? `${name}，答错，扣${scoreToSpeech(-delta)}分。`
        : `${name}，答错，本题零分。`;
      break;
    case 'timeout':
      text = delta < 0
        ? `${name}，超时，扣${scoreToSpeech(-delta)}分。`
        : `${name}，超时，本题零分。`;
      break;
    case 'violation':
      text = `${name}，违规，扣${scoreToSpeech(-delta)}分。`;
      break;
    case 'spot':
      text = `${name}，找到${scoreToSpeech(delta)}处，得${scoreToSpeech(delta)}分。`;
      break;
    case 'flower_valid':
      text = `${name}，有效，得一分。`;
      break;
    case 'flower_winner':
      text = `${name}，擂主，额外得三分。`;
      break;
    case 'adjust':
      text = delta >= 0
        ? `${name}，手动加${scoreToSpeech(delta)}分。`
        : `${name}，手动扣${scoreToSpeech(-delta)}分。`;
      break;
  }

  if (text) speak(text, { priority: 'P2' });  // P2 优先级，打断读题但不打断违规播报
}
```

`announceScore(event)` 在每次 `logEvent()` 写入 history 后自动调用，无需手动触发。


### 12.6 朗读触发机制

朗读统一由控制台（index.html）驱动，通过钩子绑定到操作事件：

```js
// 切题自动读题（第一/三环节）
function onQuestionChanged(q) {
  if (!state.tts.enabled || !state.tts.autoRead) return;
  const segments = buildQuestionSegments(q);  // 构建朗读段落列表
  speakQueue(segments);
}

// 状态变化钩子（onChange 回调内判断）
onChange(() => {
  checkAndTriggerTTS();  // 每次状态变化后检查是否需要触发新播报
});
```

**触发优先级**（高优先级打断低优先级）：

| 优先级 | 场景 |
|--------|------|
| P1（最高） | 违规判定、抢答成功提示 |
| P2 | 语音报分（答对/答错/得分/扣分，见 12.5） |
| P3 | 倒计时提示 |
| P4 | 题目正文朗读 |
| P5（最低） | 环节过渡语 |

高优先级调用 `synth.cancel()` 打断当前朗读，低优先级排队等待。

### 12.7 计时器与 TTS 联动

```
[主持人点击"出题/开始计时"]
    ↓
系统开始读题（speakQueue 读题目 + 选项）
    ↓
读题完毕（onend 回调）→ 自动启动倒计时
    ↓
计时器运行，到达 countdownAt 节点时插入播报
    （如剩余10秒、5秒、3/2/1秒）
    ↓
倒计时归零 → 播报"时间到" → 自动判超时
```

**第三环节完整朗读与抢答判定流程**：

```js
/**
 * 主持人点击"出题"后调用
 * 1. 进入 reading 状态（此时按抢答器=违规）
 * 2. TTS 读完题目 + 选项后，自动说"开始抢答"
 * 3. "开始抢答"说完的 onend 回调里切换到 armed 状态，大屏同步显示横幅
 */
function r3StartQuestion(q) {
  const segments = buildQuestionSegments(q);  // 题干 + 选项文本数组

  // 进入 reading 状态，记录当前题目文本（违规后重读用）
  state.r3.buzzState = 'reading';
  state.r3.currentReadText = segments.join('');
  save();

  // 开始朗读，最后一段说"开始抢答"，onend 触发 armed
  speakQueue([...segments, '开始抢答。'], () => {
    state.r3.buzzState = 'armed';
    save();
    // 大屏收到 buzzState='armed' 后立即显示"开始抢答"横幅（onChange 驱动）
  });
}

/**
 * 违规抢答：打断 TTS → 播报违规 → 重读题目 → 恢复 reading 状态
 */
function r3EarlyBuzz(teamId) {
  synth.cancel();  // 打断当前读题
  applyTeamScore(teamId, 'r3', -2);
  logEvent({ round: 3, teamId, memberIdx: null, delta: -2, reason: 'violation' });
  announceScore(state.history[state.history.length - 1]);

  const teamName = getTeamName(teamId);
  // 播报违规后重新读题，让其他队再听一遍
  speakQueue(
    [`${teamName}，抢答违规，扣两分。`, ...buildQuestionSegments(currentR3Question()), '开始抢答。'],
    () => {
      // 重读完毕，再次进入 armed
      state.r3.buzzState = 'armed';
      save();
    }
  );
}

/**
 * 合法抢答成功（armed 状态下第一个按键的队伍）
 */
function r3TryBuzz(teamId) {
  if (state.r3.buzzState !== 'armed') return;
  synth.cancel();  // 打断"开始抢答"残余播报（如有）
  state.r3.buzzState = 'locked';
  state.r3.buzzedTeam = teamId;
  state.r3.selectedTeam = teamId;
  state.r3.selectedMember = null;
  state.r3.buzzPulse = (state.r3.buzzPulse || 0) + 1;
  state.r3.lastBuzzTeam = teamId;
  save();
  // 大屏显示"XX队 抢答成功"（onChange 驱动）
  // TTS 播报
  speak(`${getTeamName(teamId)}，抢答成功，请答题。`);
  // 启动 15 秒答题计时
  startTimer(15, 3, true);
}
```

## 十三、抽签功能设计

### 13.1 抽签场景与时机

| 抽签类型 | 触发时机 | 决定内容 | 影响环节 |
|---------|---------|---------|---------|
| 队伍出场顺序 | 赛前准备阶段 | 5 队的参赛顺序（第 1~5 出场） | 第一、四、五环节 |
| 识图找茬图片分配 | 赛前准备阶段 | 每队对应哪张场景图 | 第四环节 |

两次抽签均在赛前 6 分钟准备阶段完成，主持人操作控制台，结果同步推送到大屏公示，观众现场见证。

---

### 13.2 抽签流程

#### 队伍出场顺序抽签

```
主持人点击 [开始抽签] 按钮
    ↓
大屏进入「抽签动画」模式
    - 5 个队名卡片在屏幕上随机滚动洗牌（持续约 3~4 秒）
    ↓
动画停止，结果落定
    - 大屏展示：1号位 → 队X，2号位 → 队Y，……
    - 控制台同步显示结果
    ↓
主持人可选择 [重新抽签]（动画再次播放）
    ↓
主持人点击 [确认结果]
    - draw.isCompleted = true（出场顺序部分锁定）
    - 大屏显示最终顺序确认画面，保留约 10 秒
```

#### 识图找茬图片分配抽签

```
主持人点击 [图片分配抽签] 按钮
    ↓
大屏进入「图片抽签动画」模式
    - 5 张图以「信封/卡牌翻转」效果随机分配给 5 队（依次揭晓）
    ↓
结果落定：图1→队A，图2→队B，……
    ↓
主持人可选择 [重新抽签]
    ↓
主持人点击 [确认结果]
    - draw.r4ImageMap 写入最终映射关系，锁定
```

两次抽签均确认后，界面显示「抽签完成」状态，所有抽签按钮禁用，进入正式比赛。

### 13.4 核心函数

```js
/**
 * 随机打乱 5 队出场顺序（Fisher-Yates shuffle）
 * 写入 state.draw.teamOrder，广播大屏触发动画
 */
function drawTeamOrder() {
  if (state.draw.orderLocked) return;
  const ids = state.teams.map(t => t.id);
  for (let i = ids.length - 1; i > 0; i--) {
    const j = Math.floor(Math.random() * (i + 1));
    [ids[i], ids[j]] = [ids[j], ids[i]];
  }
  state.draw.teamOrder = ids;
  state.draw.log.push({ type: 'order', result: [...ids], ts: Date.now() });
  save();
}

/**
 * 随机将图片序号 0~4 分配给 5 支队伍
 * 写入 state.draw.r4ImageMap，广播大屏触发动画
 */
function drawR4Images() {
  if (state.draw.imageLocked) return;
  const imgIdxs = [0, 1, 2, 3, 4];
  for (let i = imgIdxs.length - 1; i > 0; i--) {
    const j = Math.floor(Math.random() * (i + 1));
    [imgIdxs[i], imgIdxs[j]] = [imgIdxs[j], imgIdxs[i]];
  }
  const map = {};
  state.teams.forEach((t, i) => { map[t.id] = imgIdxs[i]; });
  state.draw.r4ImageMap = map;
  state.draw.log.push({ type: 'image', result: { ...map }, ts: Date.now() });
  save();
}

/** 锁定出场顺序（确认后不可重抽） */
function lockDrawOrder() {
  if (!state.draw.teamOrder.length) return false;
  state.draw.orderLocked = true;
  save();
  return true;
}

/** 锁定图片分配（确认后不可重抽） */
function lockDrawImages() {
  if (!Object.keys(state.draw.r4ImageMap).length) return false;
  state.draw.imageLocked = true;
  save();
  return true;
}

/** 解锁重抽（仅在 clearAllScores 时调用） */
function unlockDraw() {
  state.draw = { teamOrder: [], r4ImageMap: {}, orderLocked: false, imageLocked: false, log: [] };
  save();
}

/** 手动覆盖出场顺序（拖拽排序结果写入） */
function setTeamOrderManual(orderArr) {
  if (state.draw.orderLocked) return;
  state.draw.teamOrder = orderArr;
  state.draw.log.push({ type: 'order_manual', result: [...orderArr], ts: Date.now() });
  save();
}

/** 手动指定某队对应图片（下拉选择覆盖） */
function setR4ImageManual(teamId, imgIdx) {
  if (state.draw.imageLocked) return;
  state.draw.r4ImageMap[teamId] = imgIdx;
  state.draw.log.push({ type: 'image_manual', teamId, imgIdx, ts: Date.now() });
  save();
}
```

### 13.7 防错与约束

| 场景 | 处理方式 |
|------|---------|
| 未抽签就进入环节 | 进入第一/四/五环节时检测 `draw.teamOrder` 是否非空；为空则弹出警告，阻止继续 |
| 图片分配有重复 | 手动模式下，提交前校验所有队图片编号唯一；若重复则提示并阻止锁定 |
| 抽签后队名被修改 | 队名修改不影响 `teamOrder`（按 id 存储），UI 动态读取最新 name 显示 |
| 误操作重抽 | 锁定后按钮禁用；「清空所有数据」时才同步解锁，并在弹窗中明确提示「将同时重置抽签结果」 |
| 比赛中途设备刷新 | 抽签结果已持久化到 localStorage，刷新后自动恢复 |

### 13.8 强制覆盖出场顺序（应急暗门）

#### 背景

正常流程下，抽签锁定后 `draw.orderLocked = true`，所有修改入口均禁用。但现场可能出现例外情况（设备故障、选手临时缺席等），评委要求调换两队出场位置，若走完整解锁→重抽→重新确认流程，会耽误赛程、引发混乱。

为此，在设置面板提供一个**极高权限的强制覆盖入口**，绕过锁定直接改写 `draw.teamOrder`，并在流水中留存操作记录供事后审计。

#### 入口设计（故意设置操作门槛）

入口位于设置面板最底部「高级 / 危险操作」区，默认折叠，需**连续点击区域标题 3 次**才展开，防止误触：

```
┌──────────────────────────────────────────────────────────┐
│  ⚠️ 应急操作（点击标题 3 次展开）                           │
│  ──────────────────────────────（折叠，默认不可见）─────── │
│                                                          │
│  强制覆盖出场顺序                                         │
│  此操作绕过抽签锁定，直接修改出场顺序，请谨慎使用。          │
│                                                          │
│  操作人备注（必填）：[________________]                    │
│                                                          │
│  当前顺序：红队 · 绿队 · 蓝队 · 黄队 · 紫队               │
│                                                          │
│  新顺序（拖拽排序或下拉框选择）：                           │
│  1号位[红队▼]  2号位[蓝队▼]  3号位[绿队▼]  4号位[黄队▼]  5号位[紫队▼]│
│                                                          │
│  [取消]                       [强制覆盖（不可撤销）]       │
└──────────────────────────────────────────────────────────┘
```

**操作门槛**：
1. 需连续点击折叠区标题 3 次才展开（计数 5 秒内重置）
2. 「操作人备注」为必填项，为空时「强制覆盖」按钮禁用
3. 点击「强制覆盖」后弹出二次确认对话框：
   > 确认强制覆盖出场顺序？此操作将绕过抽签锁定，且不可撤销。请确保已获评委组授权。

#### 核心函数

```js
/**
 * 强制覆盖出场顺序（绕过锁定）
 * @param {Number[]} newOrder  新的 team.id 顺序数组
 * @param {String}  operator  操作人备注（必填，写入 log）
 */
function forceOverrideTeamOrder(newOrder, operator) {
  if (!operator || !operator.trim()) {
    throw new Error('操作人备注不能为空');
  }
  if (newOrder.length !== state.teams.length) {
    throw new Error('队伍数量不匹配');
  }
  // 校验：newOrder 必须是 team.id 的全排列
  const validIds = new Set(state.teams.map(t => t.id));
  const inputIds = new Set(newOrder);
  if (![...validIds].every(id => inputIds.has(id))) {
    throw new Error('出场顺序包含无效的队伍 ID');
  }

  const prev = [...state.draw.teamOrder];
  state.draw.teamOrder = newOrder;
  // 注意：不修改 orderLocked，锁定状态保持不变
  state.draw.log.push({
    type: 'force_override',
    prev,
    result: [...newOrder],
    operator: operator.trim(),
    ts: Date.now(),
  });
  save();
}
```

#### 对进行中环节的影响

`draw.teamOrder` 被覆盖后，**各环节的 `currentTeamIdx` 保持不变**，下一次推进（`advanceTeam()`）时自然读取新顺序中的对应位置。

| 场景 | 行为 |
|------|------|
| 强制覆盖发生在环节开始前 | 新顺序从头生效，无影响 |
| 强制覆盖发生在第一环节进行中 | 已出场队伍不受影响，**从下一支未出场队伍起**按新顺序进行 |
| 强制覆盖发生在第四环节进行中 | 同上，已找茬完的队不受影响 |
| 强制覆盖发生在第五环节进行中 | `r5.teamOrder` 是由 `draw.teamOrder` 在环节开始时复制的独立副本，**飞花令轮次顺序不受覆盖影响**（飞花令有独立的轮次逻辑） |

> **注意**：第五环节轮次顺序（`r5.teamOrder`）在环节开始时已从 `draw.teamOrder` 复制为独立副本，强制覆盖 `draw.teamOrder` 不会改变飞花令当前令题的轮次。如需同时调整飞花令轮次，需单独提供 `r5.teamOrder` 的覆盖入口（在第五环节操作面板内）。

#### 操作留痕

强制覆盖写入 `draw.log`，「导出成绩报告」时会在附录中列出所有强制覆盖记录：

```
[强制覆盖记录]
时间：2026-06-28 14:32:11
操作人备注：红队设备故障，与蓝队换位，评委组授权
覆盖前：红队 · 绿队 · 蓝队 · 黄队 · 紫队
覆盖后：蓝队 · 绿队 · 红队 · 黄队 · 紫队
```

## 十四、翻牌选题功能

### 14.1 设计理念

**问题**：纯随机出题虽然公平，但选手无参与感，且对结果容易产生质疑（"为什么偏偏给我出这道难题"）。

**解决方案**：翻牌选题——题目预先随机打乱并隐藏在若干背面相同的"牌"里，选手自主选择翻开哪张牌，系统按选手选择揭示题目。**本质是随机，体验是自主选择**，兼顾公平性与参与感，消除争议。

**适用环节**：

| 环节 | 翻牌对象 | 说明 |
|------|---------|------|
| 第一环节（个人必答） | 题目牌 | 每位选手从剩余牌池中自选 1 张，共 40 张牌 |
| 第四环节（识图找茬） | 图片牌 | 每队翻开 1 张，揭示本队对应的场景图 |

---

### 14.2 核心机制

**牌的本质**：每张牌是一个壳，背面一样，正面绑定一道题（或一张图/一个令题主题）。牌的排列在环节开始时一次性随机打乱，之后固定不变。选手选哪张、就翻哪张，内容由系统预先绑定。

```
环节开始时：
  questions（已按该环节随机打乱）
      ↓ 一对一绑定
  cards[0] cards[1] cards[2] ... cards[N]
  （全部 faceDown，背面相同，大屏网格展示）
      ↓ 选手/主持人点击某张
  cards[i].revealed = true
  → 翻牌动画 → 显示题目内容
  → TTS 朗读题目
  → 开始计时
```

---

### 14.3 各环节翻牌细节

#### 第一环节（个人必答）· 40 张牌

```
环节开始 → 生成 40 张牌（绑定 40 道题，位置随机打乱）
    ↓
大屏展示 8×5 牌面网格（或 5×8，视屏幕比例）
所有牌正面朝下，编号 1~40 仅供定位
    ↓
轮到某选手时：
  控制台显示：「请{选手名}选牌（点击牌号或由选手口报牌号）」
  大屏高亮当前选手所在队颜色
    ↓
选手口头报出牌号（如"我选 17 号"）
  → 主持人在控制台点击对应牌
    ↓
翻牌动画（约 0.8 秒）→ 正面显示题目
  → TTS 朗读题目 + 选项
  → 开始 15 秒计时
    ↓
判分后，该牌标记为已翻（半透明灰显），选手不可再选
```

**牌面视觉状态**：
- 未翻：背面（统一图案，如品牌 Logo 或"?"）
- 当前选中（动画中）：翻转过渡态
- 已翻/已用：正面朝上半透明，隐约可见题号，不显示题目内容
- 当前展示：正面朝上，高亮边框

#### 第四环节（识图找茬）· 5 张图片牌

```
大屏显示 5 张背面相同的牌（对应 5 张场景图）
（图片已由赛前抽签分配给各队，但内容未揭示）
    ↓
轮到某队时：
  该队对应的牌自动高亮（因为抽签已决定哪张是哪队的）
  队代表翻牌（主持人点击）
    ↓
翻牌动画 → 场景图全屏展示
  → TTS：「请{队名}开始找茬，计时六十秒」
  → 60 秒计时开始
```

### 14.5 核心函数

```js
/**
 * 为指定环节生成牌组（随机打乱绑定关系）
 * round: 1|4
 */
function initCardDeck(round) {
  const questions = getQuestionsForRound(round);  // 取该环节题目
  // Fisher-Yates 打乱题目顺序
  const shuffled = fisherYatesShuffle([...questions]);
  // 生成牌，位置索引即为 position
  state.cardFlip.cards = shuffled.map((q, i) => ({
    id: `r${round}_card_${String(i).padStart(2, '0')}`,
    position: i,
    faceValue: buildFaceValue(round, q),
    revealed: false,
    skipped: false,
    pickedBy: null,
  }));
  state.cardFlip.context = {
    round, teamId: null, memberIdx: null,
    pickCount: 1, picked: [],
  };
  save();
}

/**
 * 选手/主持人选中某张牌（翻开前的确认步骤）
 */
function selectCard(cardId) {
  const card = state.cardFlip.cards.find(c => c.id === cardId);
  if (!card || card.revealed || card.skipped) return false;
  state.cardFlip.context.picked = [cardId];
  save();
  return true;
}

/**
 * 确认翻牌（播放动画、绑定题目到当前答题流程）
 */
function confirmFlip(cardId) {
  const card = state.cardFlip.cards.find(c => c.id === cardId);
  if (!card || card.revealed) return false;
  card.revealed = true;
  card.pickedBy = {
    teamId: state.cardFlip.context.teamId,
    memberIdx: state.cardFlip.context.memberIdx,
    ts: Date.now(),
  };
  state.cardFlip.lastFlippedCard = card;
  state.cardFlip.flipPulse = (state.cardFlip.flipPulse || 0) + 1;
  // 将翻出的题目推入当前环节的答题流程
  activateFlippedQuestion(card);
  save();
  return true;
}

/**
 * 获取剩余可选牌（未翻）
 */
function getAvailableCards() {
  return state.cardFlip.cards.filter(c => !c.revealed);
}
```

### 15.2 计时核心设计

**原则**：用 `Date.now()` 时间戳而非递减变量驱动计时，两个窗口（控制台 + 大屏）各自用 `requestAnimationFrame` / `setInterval` 独立渲染，不依赖跨窗口同步每一帧——只同步"启动/暂停/结束"事件，避免 BroadcastChannel 延迟导致大屏倒计时飘移。

```
剩余时间 = durationMs − (Date.now() − startedAt) − elapsedMs（已暂停时长）
```

**状态转移**：

```
idle
 ├─ [startTimer()] ──────────────────→ running
running
 ├─ [pauseTimer()] ──────────────────→ paused
 ├─ 剩余时间 ≤ 0 ──────────────────→ expired → 触发 onExpire(round)
 └─ [resetTimer()] ──────────────────→ idle
paused
 ├─ [resumeTimer()] ─────────────────→ running
 └─ [resetTimer()] ──────────────────→ idle
expired
 └─ [resetTimer()] ──────────────────→ idle
```

### 15.3 核心函数（shared.js）

```js
/**
 * 启动倒计时
 * @param {number} durationSec  限时秒数
 * @param {number} round        当前环节（超时回调用）
 * @param {boolean} autoExpire  到 0 是否自动触发超时判定
 */
function startTimer(durationSec, round, autoExpire = true) {
  state.timer = {
    state: 'running',
    durationMs: durationSec * 1000,
    startedAt: Date.now(),
    pausedAt: null,
    elapsedMs: 0,
    round,
    autoExpire,
  };
  save();
}

/** 暂停（记录已耗时长） */
function pauseTimer() {
  if (state.timer.state !== 'running') return;
  state.timer.pausedAt = Date.now();
  state.timer.elapsedMs += state.timer.pausedAt - state.timer.startedAt;
  state.timer.startedAt = null;
  state.timer.state = 'paused';
  save();
}

/** 继续（重置 startedAt） */
function resumeTimer() {
  if (state.timer.state !== 'paused') return;
  state.timer.startedAt = Date.now();
  state.timer.pausedAt = null;
  state.timer.state = 'running';
  save();
}

/** 重置（清空） */
function resetTimer() {
  state.timer = {
    state: 'idle', durationMs: 0,
    startedAt: null, pausedAt: null, elapsedMs: 0,
    round: null, autoExpire: true,
  };
  save();
}

/** 标记超时（由 UI 层检测到剩余 ≤ 0 时调用） */
function expireTimer() {
  if (state.timer.state !== 'running') return;
  state.timer.state = 'expired';
  save();
  if (state.timer.autoExpire) onTimerExpired(state.timer.round);
}

/** 实时计算剩余毫秒（纯计算，不写 state） */
function getRemainingMs() {
  const t = state.timer;
  if (t.state === 'idle' || t.state === 'expired') return 0;
  if (t.state === 'paused') return Math.max(0, t.durationMs - t.elapsedMs);
  return Math.max(0, t.durationMs - t.elapsedMs - (Date.now() - t.startedAt));
}
```

### 15.4 超时回调（onTimerExpired）

```js
function onTimerExpired(round) {
  switch (round) {
    case 1:
      // 个人必答超时：0 分，不扣分，自动进入下一位选手
      scoreR1(state.r1.currentTeam, state.r1.currentMemberIdx, 'wrong');
```

### 15.5 UI 渲染（控制台 + 大屏共用逻辑）

两个页面各自用 `setInterval(tick, 100)` 刷新显示，读取 `getRemainingMs()` 渲染：

```js
let countdownInterval = null;

function startCountdownUI() {
  clearInterval(countdownInterval);
  countdownInterval = setInterval(() => {
    const rem = getRemainingMs();
    renderTimer(rem);
    // 检测倒计时播报节点（仅控制台调用，大屏不播报 TTS）
    checkTTSCountdown(rem);
    // 检测是否到 0
    if (rem <= 0 && state.timer.state === 'running') {
      clearInterval(countdownInterval);
      expireTimer();
    }
  }, 100);
}

function renderTimer(remainingMs) {
  const sec = Math.ceil(remainingMs / 1000);
  const pct = remainingMs / state.timer.durationMs;  // 0~1
  // 更新数字显示
  timerEl.textContent = sec;
  // 更新进度条宽度
  timerBar.style.width = `${pct * 100}%`;
  // 颜色阶段
  timerEl.className = sec <= 3 ? 'danger' : sec <= 10 ? 'warn' : 'ok';
}
```

| > 10 秒 | 绿色 | 绿色 | 无 |
| 6~10 秒 | 橙黄色 | 橙黄色 | 无 |
| 3~5 秒 | 红色 | 红色 | 无 |
| 1~3 秒 | 红色 | 红色 | 数字 scale 脉冲闪烁 |
| 0 秒 | — | 空 | 超时动画触发 |

### 15.6 TTS 倒计时播报节点

由控制台的 `checkTTSCountdown()` 检测，每个节点只播报一次（用 `lastAnnouncedSec` 去重）：

```js
const COUNTDOWN_AT = [10, 5, 3, 2, 1];  // 可在设置中自定义
let lastAnnouncedSec = -1;

function checkTTSCountdown(remainingMs) {
  if (!state.tts.enabled || !state.tts.readCountdown) return;
  const sec = Math.ceil(remainingMs / 1000);
  if (COUNTDOWN_AT.includes(sec) && sec !== lastAnnouncedSec) {
    lastAnnouncedSec = sec;
    tts.speak(`还剩${sec}秒。`);
  }
  if (remainingMs <= 0 && lastAnnouncedSec !== 0) {
    lastAnnouncedSec = 0;
    tts.speak('时间到。');
  }
}
```

### 15.10 防错与边界

| 场景 | 处理方式 |
|------|---------|
| 计时运行中切换环节 | `switchRound()` 内强制调用 `resetTimer()`，防止旧计时器残留 |
| 计时运行中设备刷新 | `startedAt` 已持久化，刷新后 `getRemainingMs()` 仍能正确计算剩余时间；若刷新耗时超过剩余时间则直接进入 expired |
| 主持人手动提前结束（选手抢先答对） | 调用 `resetTimer()` 后再判分，不触发超时回调 |
| 多次快速点击「开始计时」 | `startTimer()` 先调用 `resetTimer()` 清空旧状态再启动，不叠加 |
| 暂停期间 TTS 正在播报倒计时 | 暂停后 `checkTTSCountdown` 停止检测；恢复后重置 `lastAnnouncedSec` 防止漏播 |
